<objective>
Apply the desktop-edit width fix uniformly across all five entity forms per D-13/D-14/D-15/D-21. Each row gets two Edit buttons (mobile + desktop, differentiated by Tailwind responsive visibility). The server's per-router `_hydrate_form_context` (or equivalent helper) reads the `?layout=desktop` query param and wires `form_target` + `form_swap` accordingly. The form template uses `{{ form_target }}` and `{{ form_swap }}` as context-driven values, NOT hardcoded strings. On desktop edit success, the response uses OOB swaps: updated row by id + form mount cleared.

Purpose: Closes the residual width bug that the commit a3a2f76 Save/Cancel fix surfaced but didn't fully close. Mount-based rendering for desktop avoids foster-parenting a `<div>` into table context — the root cause of the narrow-column bug.
Output: Five entity row templates with dual buttons. Five entity form templates with context-driven form_target/form_swap. Five routers' edit + update handlers reading the layout query param and a hidden form input. New OOB row+mount response pattern on desktop edit success.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client→GET /{entity}/{id}/edit?layout=desktop | Authenticated user requests the edit form variant; layout is a query string param (untrusted user input) |
| client→POST /{entity}/{id} with hidden layout input | Authenticated user submits an edit; layout is a hidden form field (untrusted user input); CSRF-protected |
| Server OOB swap response | Server returns HTML with hx-swap-oob attributes; browser HTMX layer must trust the response (TLS in transit + same-origin) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
