## Multi-origin + varietal data shape

### Q1: How should multi-origin be modeled for CATALOG-03?

| Option | Description | Selected |
|--------|-------------|----------|
| Join table: coffee_origins | New table (id, coffee_id, country, region NULL, percent NULL, sort_order). Most flexible. | |
| ARRAY column: origins TEXT[] | Single column on coffees; is_blend = len > 1. Simplest migration. | |
| is_blend bool + origins TEXT[] | Same as array but explicit is_blend column. | |
| Other (free text) | User answer: "What do you think is best for versatility and future growth" | ✓ |

**User's choice:** Asked Claude to recommend for versatility + future growth.
**Claude's recommendation (accepted in Q5 follow-up):** Join table `coffee_origins`. Reasoning: symmetry with the m2m varietal choice they’d pick next, headroom for region granularity / per-origin tasting notes / optional %-share later, comparable migration cost to the array option, slightly more code but consistent with existing service patterns.

## Flavor-note bidirectional sync semantics

### Q1: Should the brew session UI distinguish 'inherited' chips from 'added-by-user' chips visually?

| Option | Description | Selected |
|--------|-------------|----------|
| No -- treat all chips identically | Simpler UI; kettle-side cognitive minimization. | ✓ |
| Subtle visual hint | Inherited chips outlined / icon. | |
| Strong badge | "(from coffee)" label on inherited chips. | |

**User's choice:** Treat identically.

### Q2: When does write-back to the parent coffee happen?

| Option | Description | Selected |
|--------|-------------|----------|
| On session save only | Diff session’s final chips vs coffee’s current on save; apply union/difference. | ✓ |
| Live on each chip add/remove | Immediate write per chip change. | |

**User's choice:** On session save only.

### Q3: Should write-back add AND remove, or only add?

| Option | Description | Selected |
|--------|-------------|----------|
| Add AND remove | Literal CATALOG-06; both directions sync. | ✓ |
| Add only (one-way enrichment) | Session adds enrich coffee; removes don’t propagate. | |
| Add always; remove only if user explicitly opts in | Prompt "Also remove from this coffee?" on removal. | |

**User's choice:** Add AND remove.

### Q4: Cross-user conflict (coffees are household-shared)?

| Option | Description | Selected |
|--------|-------------|----------|
| Last write wins | Shared catalog philosophy already established. | ✓ |
| Track provenance per chip | Store which user added each chip. | |
| Removes only soft-tombstone the chip | Per-user prefill skip list. | |

**User's choice:** Last write wins.

### Q5 (follow-up): Draft restore behavior when advertised chips changed since draft creation?

| Option | Description | Selected |
|--------|-------------|----------|
| Draft wins -- restore exactly as left | Drafts capture user intent at a moment. | |
| Re-prefill: union of draft chips and current coffee chips | Take draft chips + any new advertised chips since. | ✓ |
| Re-prefill: current coffee chips only | Discard draft chip edits. | |

**User's choice:** Union of draft + current coffee chips.

### Q6 (follow-up): CSV import -- trigger write-back to parent coffee?

| Option | Description | Selected |

### Q7 (follow-up): Prefill source beyond `coffees.advertised_flavor_note_ids`?

| Option | Description | Selected |
