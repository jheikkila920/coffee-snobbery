## Special Pattern Notes

### 1. HTMX repeating-row `+ Add another` pattern — FIRST USE

No existing analog in this codebase. Recommended approach is HTMX Option A (server-rendered partial append):

- `GET /coffees/origin-row-template?index=N` returns one `coffee_origin_row.html` fragment with `removable=True`
- Router passes `index` (for future sort_order) and returns the fragment (no auth required — this is a GET for static HTML scaffolding, but apply `require_user` for defense in depth)
- The `+ Add another origin` button uses `hx-get="/coffees/origin-row-template"` + `hx-target="#origin-rows-container"` + `hx-swap="beforeend"`
- Remove button uses `hx-on:click="this.closest('[data-origin-row]').remove()"` — vanilla DOM, CSP-compliant via HTMX 2.x `hx-on:` syntax
- First row is rendered with `removable=False` to prevent removing the last origin

### 2. Pydantic `CoffeeCreate` restructuring — wire format for `origins[]`

`origins_country[]` and `origins_region[]` are NOT Pydantic fields. The router handles them via:
```python
origins_country = form_data.getlist("origins_country")
origins_region = form_data.getlist("origins_region")
origins = list(zip(origins_country, origins_region))
```
This is analogous to how `advertised_flavor_note_ids` bypasses Pydantic (handled in `_parse_form_payload` separately, then passed directly to the service). The schema's `extra="forbid"` is satisfied because these keys never reach the schema.

### 3. OOB swap on desktop edit success — FIRST USE for update path

The existing `hx-swap-oob` in this codebase is used ONLY for:
- Form mount clear after create: `<div id="X-form-mount" hx-swap-oob="innerHTML"></div>`
- List refresh after equipment create: `<div id="equipment-list" hx-swap-oob="innerHTML">`
- Admin backup list refresh

Desktop edit success requires a NEW pattern: OOB row replacement by element ID + mount clear. The response can either be structured as the primary swap targeting `#entity-form-mount` (innerHTML → empty) with the row update as OOB, or vice versa. Recommended: make the row the OOB element (identified by `id="coffee-{id}"`) and the mount-clear the primary target of the form's `hx-target`.

### 4. Mode-aware form coverage (D-14 uniformity check)

All five entity forms already use `{% set is_edit = mode == "edit" %}`. The key change is:
- `coffee_form.html`: currently hardcodes `form_target`/`form_swap` in the template → must be replaced with server-injected context variables (from updated `_hydrate_form_context`)
- `roaster_form.html`: already uses computed Jinja vars for `form_target`/`form_swap` — the extension just adds `layout` param input from the server
- `equipment_form.html`, `flavor_note_form.html`, `recipe_form.html`: same pattern as roaster — currently hardcode `form_target` in template, must switch to server-injected vars

The pattern is: server computes `form_target`/`form_swap` based on `mode` + `layout`, injects into template context. Template uses `{{ form_target }}` and `{{ form_swap }}` directly in `hx-target` and `hx-swap` attributes. Cancel buttons use the same context variables.
