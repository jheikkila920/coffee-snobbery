# Phase 15.1: Catalog & Session Polish - Context

**Gathered:** 2026-05-26
**Status:** Ready for planning

<domain>
## Phase Boundary

Clean up the shared coffee catalog data model and brew-session prefill so all downstream v1.2 feature work builds on a clean schema. Seven locked requirements (CATALOG-01..07):

- **CATALOG-01:** Edit-coffee modal width parity with create-coffee on desktop. (Save/Cancel inertness already fixed in commit `a3a2f76`; this phase closes the residual width issue.)
- **CATALOG-02:** Drop `coffees.country`; merge surviving data into the new origin shape.
- **CATALOG-03:** Single-origin vs. blend; multi-origin as structured rows, not free-text.
- **CATALOG-04:** Roast-level enum extended with Nordic Light and Ultra Light at the lighter end.
- **CATALOG-05:** Varietal stays optional; multi-select with autocomplete from a seeded catalog.
- **CATALOG-06:** New brew session inherits flavor notes from the parent coffee; user can remove inherited chips; session-level adds/removes mirror back to the parent coffee (bidirectional).
- **CATALOG-07:** Roast-freshness tracking removed app-wide -- column dropped, AI prompts cleaned, scheduler/UI/analytics scrubbed.

Explicitly NOT in this phase:
- Any IA / nav redesign (Phase 17 owns).
- Any mobile visual rework beyond the edit-form width fix (Phase 21 owns).
- New `cafe_logs` schema or anything CAFE-* (Phase 16 owns; Phase 15.1 cleans the surface 16 builds on).
- AI page consolidation, AI research/predict, charts (Phase 19 owns).
- Roast-level UI styling changes -- enum value addition only, no visual revamp.

</domain>

<decisions>
## Implementation Decisions

### CATALOG-01 -- Edit-form width fix
- **D-13:** Desktop edit renders into the existing `#entity-form-mount` wide div (the same one create uses), not in-place via `closest [data-row]`. The narrow-column bug is rooted in foster-parenting a `<div>` into table context. Mount-based rendering avoids that entirely.
- **D-14:** Apply the fix to all five entity forms uniformly -- coffees, roasters, equipment, flavor_notes, recipes. Same root cause; same fix; one shared pattern. Mobile (card layouts) keeps the existing inline `closest [data-row]` swap that the recent commit `a3a2f76` repaired -- it works correctly on `<div>` cards.
- **D-15:** Re-clicking Edit on a different row while a form is open silently overwrites the form via HTMX `innerHTML` swap. No dirty-state prompt. Same behavior as the existing create-form flow.

### CATALOG-02 / CATALOG-03 -- Multi-origin data shape
- **D-01:** Multi-origin is modeled as a new join table `coffee_origins (id BIGINT PK, coffee_id BIGINT FK -> coffees ondelete=CASCADE, country TEXT NOT NULL, region TEXT NULL, sort_order INT NOT NULL)`. `is_blend` is **derived** from `COUNT(coffee_origins) > 1` -- no explicit `is_blend` column on `coffees`.
- **D-02-orig:** No `percent` column on `coffee_origins` in v1 -- roasters typically hide blend ratios. Sum-to-100 validation would be too strict. Add later if a use case emerges.
- **D-05:** Country -> origin merge migration: one `coffee_origins` row per existing coffee. `country` = `coffees.country` if set, else `coffees.origin`. `region` = `NULL` (the existing `origin` free-text often holds a region like "Yirgacheffe" but parsing it is too risky on inconsistent data; users edit manually). `sort_order = 0`. Then drop both `coffees.country` and `coffees.origin` columns in the same migration. Origin field "wins" semantically when both columns are populated -- the more specific value carries forward.

### CATALOG-04 -- Roast-level enum widening
- **Claude's discretion** (planner's call): extend the existing `coffees_roast_level_check` CHECK constraint with two new values placed at the lighter end -- ordering for UI dropdown is `ultra-light`, `nordic-light`, `light`, `medium-light`, `medium`, `medium-dark`, `dark`, `unknown`. No data migration required (existing values still valid). Forward-compatible.

### CATALOG-05 -- Varietal model
- **D-02:** Varietals are a many-to-many: new table `varietals (id BIGINT PK, name CITEXT NOT NULL UNIQUE, created_at)` + join table `coffee_varietals (coffee_id, varietal_id)`. CITEXT + UNIQUE collapses "Geisha"/"geisha"/"GEISHA" to one row. Symmetric with the `coffee_origins` shape and with existing `flavor_notes` (which is also m2m-via-array; varietal can either follow that pattern or use a true join table -- planner's call which is more consistent, but a true join table is preferred for query simplicity and so that varietal-level analytics can index cleanly later).
- **D-03:** Any authenticated user can create a new varietal on the fly from the autocomplete UI -- type a name not in the list, get an "Add 'X' as a new varietal?" affordance, insert and select. Mirrors the existing flavor_notes UX (Phase 4). Citext UNIQUE prevents trivial dupes.
- **D-04:** Seed the `varietals` table with ~12-15 common varietals via the migration's `op.bulk_insert`: Bourbon, Typica, Caturra, Catuai, Geisha, Pacamara, SL28, SL34, Mundo Novo, Pacas, Heirloom, Maragogype, Castillo, Catimor. Lean seed; users add the rest.

### CATALOG-06 -- Brew session flavor-note inheritance + bidirectional sync
- **D-06:** New brew session prefills `flavor_note_ids_observed` with the parent coffee's `advertised_flavor_note_ids`. Inherited chips and user-added chips are visually **identical** in the form -- no badge, no outline variation. Simpler kettle-side UX.
- **D-07:** Write-back to `coffees.advertised_flavor_note_ids` happens **on session save only**, not on each chip add/remove. The save handler diffs the session's final `flavor_note_ids_observed` against the coffee's current `advertised_flavor_note_ids` and applies the union/difference. Plays nicely with `brew_drafts` (server-side autosave-on-blur) -- drafts never touch the parent coffee.
- **D-08:** Sync is bidirectional: session-level adds **and** removes are mirrored to the parent coffee. Literal CATALOG-06 wording. If user removes an inherited chip on a session, that chip is removed from `coffees.advertised_flavor_note_ids`.
- **D-09:** Cross-user behavior is **last-write-wins**, consistent with the shared-catalog philosophy already established (any user can edit any coffee). If User A removes a chip that User B added via a session, User B's next prefill for that coffee no longer includes it. Document the behavior in a UI tooltip on the chip section.
- **D-10:** Draft restore = **union of draft chips + current coffee chips**. When a user reopens a `brew_drafts` row for a coffee whose advertised chips have changed since the draft was created, prefill is `set(draft.flavor_note_ids_observed) UNION set(coffees.advertised_flavor_note_ids)`. New advertised chips show up; removed advertised chips don't disappear from the draft. Predictable additive merge.
- **D-11:** CSV import of brew sessions triggers the same write-back semantics as the UI -- imported rows enrich the parent coffee's advertised chips identically. Symmetric. Risk is small because the CSV import already validates entity existence per Phase 5.
- **D-12:** Prefill source is **only** `coffees.advertised_flavor_note_ids`. Do not also pull the user's most-recent session chips for the same coffee -- the existing prefill summary surface already shows those separately. Keep the chip prefill source single and predictable.

### CATALOG-07 -- Roast-freshness removal
- **D-16:** Drop `bags.roast_date` outright in a single Alembic migration with **no preservation**. Nightly `pg_dump` backups are the recovery story if the data is ever needed. Acceptable because (a) the feature is being removed on purpose, (b) household-scale, (c) backups already cover it.
- **D-17:** AI cached recommendations are invalidated by the signature change naturally -- removing `Bag.roast_date.isoformat()` from `analytics.py` per-coffee signature means the **next nightly @ 00:00 cadence sees a mismatch and regenerates** for every user with an AI key. No force-regen, no immediate wipe, no startup job. Trust the signature-based regen design.
- **D-18:** Strip every freshness mention from the AI prompt(s) in `app/services/ai_service.py` and any prompt-builder modules -- grep for "freshness", "roast date", "days off roast", "resting". Removal is mechanical; the prompt audit is a planner QA gate -- add a one-line "AI service file has zero references to freshness/roast_date" assertion to the plan's must_haves.
- **D-19:** Remove `freshness_buckets()` from `app/services/analytics.py` entirely, plus the home-page fragment `app/templates/fragments/home/roast_freshness.html`, the router call in `app/routers/home.py`, and any analytics module export. Update tests; remove the freshness fixture if any. Also drop the "Pitfall 4 (reads bags.roast_date ONLY)" comment in the analytics module since the column is gone.

### Claude's Discretion
- Exact CHECK constraint syntax for CATALOG-04 enum widening (drop + recreate vs `op.execute("ALTER TABLE ... DROP CONSTRAINT ... ADD CONSTRAINT ...")`) -- planner's call.
- Whether varietals follows the existing `flavor_notes`-style array-on-parent pattern or a true `coffee_varietals` join table -- a true join table is preferred for varietal-level analytics later (Phase 19) and for query simplicity, but if planner's research shows a strong reason to mirror the array pattern, that's acceptable as long as autocomplete + add-on-the-fly + citext-collapsed uniqueness all hold.
- The desktop-vs-mobile routing mechanism for the new edit form mount -- a query param (`?layout=desktop`), a viewport header sniff, or a template-level branch in the list pages that emits different `hx-target` per row. Planner's call as long as D-13/D-14/D-15 hold and the existing `closest [data-row]` mobile path stays unbroken.
- Migration ordering between CATALOG-02 (drop country) and CATALOG-03 (introduce coffee_origins). One migration per requirement vs. one combined migration -- both are acceptable. Combined is simpler if the data move and column drops happen in one atomic transaction; planner's call.
- Whether the inline "add new varietal" affordance lives in the same fragment as the autocomplete or as a separate small HTMX POST -- planner's call.

</decisions>

## Existing Code Insights

### Reusable Assets
- **Inline-edit Save/Cancel mechanism is already fixed** in commit `a3a2f76` across all five entity forms (`coffee_form.html`, `roaster_form.html`, `equipment_form.html`, `flavor_note_form.html`, `recipe_form.html`) plus the matching `GET /{id}/row` endpoints. Phase 15.1 keeps that mechanism for **mobile** card layouts and adds the desktop mount-based rendering on top -- it does NOT rip out the row-based pattern.
- **`advertised_flavor_note_ids` BIGINT[] + GIN index** in `app/models/coffee.py` -- existing pattern for "many flavor notes per coffee, indexed for containment queries." The write-back diff logic in CATALOG-06 lands in service code that already knows how to mutate this column. SQLAlchemy 2.0 + Alembic autogenerate cannot emit `USING GIN` -- pattern is to hand-edit the migration to add it via `op.execute()`. Same pattern applies if any new array column is added.
- **`flavor_notes` "create on the fly from autocomplete"** UX pattern in `app/services/flavor_notes.py` + `app/templates/fragments/flavor_note_form.html` + the autocomplete fragment in `coffee_form.html` -- the new varietal autocomplete (D-03) should mirror this API surface.
- **Signature-based AI regen** in `app/services/analytics.py` (signature builder at line 371-411) + `app/services/ai_service.py` -- the design that makes D-17 work without an explicit regen job. Removing `roast_date.isoformat()` from the signature is a one-token change that triggers natural regen via the nightly cadence.
- **Pydantic CoffeeCreate reused for create + edit** in `app/schemas/coffee.py` -- one schema must handle both flows; restructure carefully so `origins[]` and `varietal_ids[]` don't break edit validation when the payload shape changes.

### Established Patterns
- **CITEXT + NOT UNIQUE** for `coffees.name` (different roasters legitimately share names). Same pattern likely applies to new `coffee_origins.country` if multiple coffees can share country (they can).
- **Foreign-key directionality asymmetry** -- `coffees.roaster_id` is `SET NULL`, `bags.coffee_id` is `RESTRICT`. New `coffee_origins.coffee_id` should be `CASCADE` (origins are owned by the coffee; deleting a coffee removes its origin rows). New `coffee_varietals.coffee_id` similarly `CASCADE`. `coffee_varietals.varietal_id` should be `RESTRICT` (don't let users delete a varietal still in use).
- **Single uvicorn worker invariant** is unchanged by this phase. No threading or worker-count assumptions to revisit.
- **CHECK constraints shaped as `IS NULL OR ...`** because both columns are nullable. CATALOG-04 enum extension preserves that shape.
- **Server-side draft autosave-on-blur** via `brew_drafts` -- D-10 union-merge logic must respect drafts as user-intent snapshots, not stale state.
- **`coffee-form-mount` div above the table** in `app/templates/pages/coffees.html:66` -- the existing create-form mount. All five entity list pages have the equivalent (e.g., `roaster-form-mount`). D-13 reuses these.
- **Mode-aware form fragments via Jinja `{% set is_edit = mode == "edit" %}` (`coffee_form.html:45`)** -- the established pattern for mode-driven form rendering; D-13's new desktop routing piggybacks on the same `mode` plumbing.

### Integration Points
- **CSV import/export** in `app/services/csv_io.py` is the connective tissue for CATALOG-02 (drop country column from CSV header), CATALOG-03 (add origins[] to CSV header), CATALOG-06 (apply write-back on imported rows -- D-11), and CATALOG-07 (drop roast_date from CSV header). Single file, four touch points.
- **Home page** (`app/routers/home.py` + `app/templates/pages/home.html`) is the integration point for CATALOG-07 removal -- the freshness card include and the router data fetch must come out together.
- **Analytics signature** (`analytics.py:371-411`) is the integration point for CATALOG-07 AI invalidation -- removing the freshness line forces the next nightly regen across all users naturally.
- **Migrations** -- one combined Alembic revision per requirement is the cleanest plan, but multiple within a single phase (e.g., one for CATALOG-02/03 combined data move, one for CATALOG-04 enum, one for CATALOG-05 varietal m2m + seed, one for CATALOG-07 column drop) is also fine. Planner's call. All must keep the **single uvicorn worker on container start** invariant unchanged.

<specifics>
## Specific Ideas

- **"Treat all chips identically" (D-06)** is a deliberate UX choice -- no inherited badge, no outline difference. Kettle-side cognitive load minimization. Tooltip on the chip section explains the bidirectional behavior once, not on every chip.
- **Last-write-wins on shared catalog (D-09)** -- the household-app philosophy. If conflict resolution ever becomes a complaint, revisit; v1.2 doesn't try to solve it.
- **Drop with no preservation (D-16)** -- conscious choice not to stage roast_date data to an archive table. The feature is being removed on purpose; pg_dump is the recovery story.
- **Let nightly cadence pick up the AI signature change (D-17)** -- trust the existing design. Don't bolt on a startup regen job.
- **Common 12-15 varietal seed (D-04)** -- specific list locked: Bourbon, Typica, Caturra, Catuai, Geisha, Pacamara, SL28, SL34, Mundo Novo, Pacas, Heirloom, Maragogype, Castillo, Catimor. Lean, covers ~95% of specialty coverage.
- **The five entity forms get the desktop width fix uniformly (D-14)** -- coffees, roasters, equipment, flavor_notes, recipes. One pattern across all.

</specifics>

<deferred>
## Deferred Ideas

- **Per-origin tasting notes / elevation / process** -- the `coffee_origins` schema supports adding these later. Not in v1.2; if needed, add columns in a forward migration.
- **Optional %-share on blend origins** -- supported by adding a `percent` column to `coffee_origins`. Deferred; revisit if a user complains that blends feel undifferentiated.
- **Region-parse from existing free-text origin** -- decided manually instead of best-effort parse. Users who care can re-enter; not worth migration complexity.
- **Per-user provenance on advertised_flavor_note_ids** (who added what chip) -- considered for cross-user conflict resolution and rejected as overkill. Last-write-wins is the policy. Revisit only if conflicts become a complaint.
- **AI startup force-regen on freshness removal** -- considered and rejected in favor of nightly cadence. Cheaper, less code.
- **`bags_roast_date_archive` table** -- considered for data preservation and rejected. pg_dump is the recovery path.
- **Dirty-state prompt on re-opening edit form (CATALOG-01)** -- considered for D-15 and rejected. Silent replace matches create-form behavior.
- **CATALOG-08 (modal/dialog pattern for catalog edits)** -- not needed. The mount-based render solves the width issue without introducing a new modal pattern. If a true modal becomes desirable for some other reason (e.g., long forms breaking page flow), that's a Phase 17 / 21 concern.
- **AI prompt audit beyond freshness** -- this phase only scrubs freshness/roast_date references. A broader prompt audit (e.g., re-tuning given multi-origin coffees and varietals) is a Phase 19 concern.

None of the above are dropped -- they are captured for the phases or future iterations that own that surface.

</deferred>
