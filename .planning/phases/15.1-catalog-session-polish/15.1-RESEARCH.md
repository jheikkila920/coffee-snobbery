## Summary

Phase 15.1 is a data model surgery phase with five independent sub-areas that share no code changes between them except the migration chain. All five sub-areas are substantially pre-decided in CONTEXT.md; research confirms the decisions against the live codebase and surfaces the integration points and landmines the planner needs.

The **biggest integration risk** is CATALOG-06 (brew-session bidirectional flavor-note sync): the write-back must hook into `create_brew_session` and `update_brew_session` in `app/services/brew_sessions.py` AND the CSV importer in `app/services/csv_io.py`, and draft restore needs a union-merge in `app/routers/brew.py`. These are three distinct touch points that must all be consistent.

The **biggest template risk** is CATALOG-02/03 (multi-origin): the `coffee_form.html` already has a deeply wired Alpine.js + HTMX chip-builder for flavor notes. Adding a repeating origin-row block with a "+ Add another origin" button requires a similar HTMX partial-append pattern without introducing Alpine x-data scope conflicts with the existing flavor-note chip component.

The **safest mechanical work** is CATALOG-07 (freshness removal): it is a pure deletion sweep. The file list is fully enumerated below with exact line numbers. No logic changes, just deletions and one migration.

**Primary recommendation:** Plan four separate migrations in order (CATALOG-02/03 combined data move, CATALOG-04 enum widening, CATALOG-05 varietal m2m + seed, CATALOG-07 column drop) so each is atomic and reversible independently. All four can execute in a single `alembic upgrade head` on container start.

**Current Edit button pattern (verified, all five entities):** Single button in every row fragment (`coffee_row.html:137`, `roaster_row.html:37`, `equipment_row.html:26/73`, etc.) uses `hx-target="closest [data-row]"` `hx-swap="outerHTML"` — the mobile pattern that `a3a2f76` confirmed works on `<div>` card mode.

**`_hydrate_form_context()` in coffees router (`coffees.py:184-236`):** Currently accepts `db`, `values`, `errors`, `mode`, `coffee_id`. Returns a dict with `values`, `errors`, `mode`, `coffee_id`, `processes`, `roast_levels`, `roaster_name`, `selected_flavor_notes`. Does NOT currently accept a `layout` parameter.

**`coffee_form.html:45-54`:** Uses `{% set is_edit = mode == "edit" %}` and hardcodes `form_target = "closest [data-row]"` for edit mode. The Cancel button at line 302 hardcodes `hx-target="closest [data-row]"` in edit mode.

### What Must Change for D-21

The form template currently hardcodes the edit target. D-21 requires it to be dynamic (desktop vs mobile path). Two approaches:

**Option A (recommended):** Pass `form_target` and `form_swap` as template context variables from `_hydrate_form_context()`. The router reads `request.query_params.get("layout")` and passes the appropriate values. The template uses `{{ form_target }}` and `{{ form_swap }}` instead of the hardcoded strings.

**Option B:** Add a second Jinja conditional `{% if layout == "desktop" %}` in the form template, passing `layout` as context.

Option A is cleaner — all five entity forms can share the exact same template pattern with zero additional logic in the template. The router injects the computed targets.

### D-21 Dual-Button Implementation

In each row template (both `mode="card"` and `mode="row"` shapes), replace the single Edit button with:

```html
{# mobile: existing pattern — hidden md:hidden on desktop #}
<button type="button"
        class="... md:hidden"
        hx-get="/{entity}/{{ entity.id }}/edit"
        hx-target="closest [data-row]"
        hx-swap="outerHTML">Edit</button>

{# desktop: new pattern — hidden on mobile #}
<button type="button"
        class="... hidden md:inline-flex"
        hx-get="/{entity}/{{ entity.id }}/edit?layout=desktop"
        hx-target="#{entity}-form-mount"
        hx-swap="innerHTML">Edit</button>
```

**HTMX 2.x compatibility confirmed:** `hx-target="closest [data-row]"` and `hx-target="#coffee-form-mount"` both use standard CSS selector syntax supported in HTMX 2.x. `hx-swap="outerHTML"` and `hx-swap="innerHTML"` are both standard. No HTMX 2 migration issues here. [VERIFIED: live codebase — both patterns already in use]

### D-13 Cancel Button on Desktop

On the desktop path, Cancel must clear the `#entity-form-mount` div (not replace `closest [data-row]`). The existing empty-form endpoints (`GET /coffees/empty-form`, etc.) already return `fragments/empty.html`. The Cancel button context must be controlled by the same `form_target`/`form_swap` variables injected from the router.

**Landmine:** The desktop Cancel must target `#{entity}-form-mount` with `innerHTML`, not `closest [data-row]`. If the same Cancel fragment tries to swap `outerHTML` on `closest [data-row]` from a form rendered inside the mount div (not inside any `[data-row]`), HTMX will travel up the DOM and fail to find a `[data-row]` ancestor, resulting in a silent no-op. The Cancel for desktop path must explicitly target the mount div.

### On Successful Edit (Desktop Path)

The existing `update_coffee_handler` success path returns `coffee_row.html` with `mode="row"` — a `<tr>`. HTMX is swapping into `#coffee-form-mount` (an inner div), not into the original table row. The success response for the desktop path needs to:
1. Return the updated row fragment to replace the correct `<tr id="coffee-{id}">` in the table — this requires an OOB swap (`hx-swap-oob`).
2. Clear the form mount (also OOB, or the row fragment includes `<div id="coffee-form-mount" hx-swap-oob="innerHTML"></div>`).

**This is the core of the "narrow column" bug from a3a2f76 revisited on success.** On the desktop edit success path, the response must include two OOB swaps: the updated row and the cleared mount. The router needs to detect `layout=desktop` on the success path (either via a hidden form field or via a referer header check) to decide which response shape to return.

**Recommendation:** Add a hidden `<input type="hidden" name="layout" value="desktop">` inside the form when rendering in desktop mode. The `update_coffee_handler` reads `form_data.get("layout")` and returns either the plain `coffee_row.html` (mobile, replaces `closest [data-row]`) or an OOB composite response (desktop, swaps the row + clears the mount).

### Migration Pattern (p4_shared_catalog.py)

The GIN-index hand-edit pattern is at `p4_shared_catalog.py:26-28`:
```python
# NOT in op.create_table() — emitted via raw op.execute AFTER table creation:
op.execute(
    "CREATE INDEX ix_coffees_advertised_flavor_note_ids ON coffees USING GIN (advertised_flavor_note_ids)"
```

The `coffee_origins` table does not need a GIN index (country is TEXT, not ARRAY). Standard B-tree on `coffee_id` is sufficient. The `coffee_origins` migration can use standard `op.create_table()` + `op.create_index()` with no hand-edit needed.

### Data Migration Steps (D-05)

```sql
-- Step 1: create coffee_origins table (before data move)
CREATE TABLE coffee_origins (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    coffee_id BIGINT NOT NULL REFERENCES coffees(id) ON DELETE CASCADE,
    country TEXT NOT NULL,
    region TEXT,
    sort_order INTEGER NOT NULL
);
CREATE INDEX ix_coffee_origins_coffee_id ON coffee_origins(coffee_id);

-- Step 2: populate from existing coffees data
INSERT INTO coffee_origins (coffee_id, country, region, sort_order)
SELECT id,
       COALESCE(NULLIF(country, ''), NULLIF(origin, ''), 'Unknown') AS country,
       NULL AS region,
       0 AS sort_order
FROM coffees
WHERE country IS NOT NULL OR origin IS NOT NULL;

-- Step 3: drop the source columns
ALTER TABLE coffees DROP COLUMN country;
ALTER TABLE coffees DROP COLUMN origin;
```

**Landmine on the data move:** `coffees` rows where BOTH `country` and `origin` are NULL will get no `coffee_origins` row. This is correct — a coffee can have unknown origin. The service and template code for the new origin display must handle `len(coffee.origins) == 0` gracefully (show "—" or nothing).

**Landmine on the country filter:** `coffees.py` currently has `list_distinct_countries()` querying `coffees.country`. After the migration this becomes a query against `coffee_origins.country` joined back. The filter bar in `pages/coffees.html:43-50` also needs to change from `name="country"` to something origin-aware.

**Alembic Migration Safety Rule:** Alembic's `--autogenerate` will detect the new `CoffeeOrigin` model and the dropped `country`/`origin` columns. The GIN index on `advertised_flavor_note_ids` is NOT in the model's `__table_args__` (the docstring in `coffee.py:100-103` explains this). Autogenerate will NOT try to drop it. Safe to use autogenerate as the starting point for this migration, then hand-add the INSERT data-move step.

### Origin-Row HTMX Pattern (D-22)

D-22 requires a repeating origin row block with "+ Add another origin" button. Two valid approaches:

**Option A (recommended): HTMX `hx-get` partial append**

A server-rendered empty origin row fragment. "+ Add another origin" fires `hx-get="/coffees/origin-row-template?index=N"` targeting a container with `hx-swap="beforeend"`. Removes require a `×` button that Alpine or vanilla JS shows on rows after the first. Server assigns row index for namespacing the form inputs.

**Option B: Alpine `x-for` over a reactive array**

Pure client-side with Alpine. `x-data="{ origins: [{ country: '', region: '' }] }"` drives `x-for`. Add/remove are Alpine methods. On form submit, origins are encoded as repeated hidden inputs.

**Option A is safer** because it does not require Alpine state management for a variable-length list, and it keeps the form inputs as standard server-rendered HTML that FastAPI's `form_data.getlist()` can collect. Option B risks Alpine initialization order issues if the form is loaded via HTMX (the `htmx:afterSettle` → `Alpine.initTree` hook handles most cases, but variable-length x-for lists have additional edge cases).

**Wire format for `origins[]` (FastAPI form-array conventions):**

FastAPI reads repeated keys via `form_data.getlist("origins_country")` etc. The cleanest convention for this codebase (matching how `advertised_flavor_note_ids` is handled) is:
```
origins_country[0]=Ethiopia&origins_region[0]=Yirgacheffe&origins_sort_order[0]=0
origins_country[1]=Kenya&origins_region[1]=&origins_sort_order[1]=1
```
FastAPI does not natively parse bracket-indexed arrays. The router must call `form_data.getlist("origins_country")`, `form_data.getlist("origins_region")`, etc. and zip them together. This is simpler than a single repeated key with JSON encoding.

**Alternative wire format:** A single `origins_json` hidden input carrying JSON (encoded by Alpine or a script). Simpler on the router side — one `json.loads()` call — but requires JS to populate it, which breaks no-JS graceful degradation. Not recommended for a form that already has a hybrid static-seed/Alpine-takeover pattern.

### Coffee-Varietal Join Table vs Array

The CONTEXT prefers a true `coffee_varietals` join table. This is confirmed as the right call:
- `advertised_flavor_note_ids` uses an array specifically because containment queries (`@>`) need the GIN index for performance. Varietals do not need containment queries in v1.
- A join table gives Phase 19 a clean `varietal_id`-indexed path for analytics without adding a GIN index later.
- CITEXT UNIQUE on `varietals.name` handles case-folding at the DB level — no need for GIN on the join table.

**Association table definition (SQLAlchemy 2.0 style):**
```python

coffee_varietals = Table(

    Base.metadata,
    Column("coffee_id", BigInteger, ForeignKey("coffees.id", ondelete="CASCADE"), primary_key=True),
    Column("varietal_id", BigInteger, ForeignKey("varietals.id", ondelete="RESTRICT"), primary_key=True),

```

## Sub-Area 4: Brew Session Flavor-Note Inheritance (CATALOG-06)

### Exact Save-Path Locations [VERIFIED]

**`create_brew_session` (`brew_sessions.py:155-215`):**
Takes `flavor_note_ids_observed: list[int]` as a plain argument. After `db.commit()` at line 213, the new write-back logic must fire. Since the session is committed before this, the write-back should be a second update within a new DB statement (or committed in the same transaction before the first commit if we restructure slightly).

**Recommended pattern:** After the session is flushed (line 212, id populated), before `db.commit()`:
1. Load `coffee.advertised_flavor_note_ids` (the current set)
2. Compute new set: `existing | set(flavor_note_ids_observed)` for adds; then handle removes relative to the coffee's current set
3. UPDATE `coffees.advertised_flavor_note_ids`
4. Single `db.commit()` covers both the session insert and the flavor-note write-back

For `update_brew_session` (`brew_sessions.py:218`+): the diff needs the session's OLD `flavor_note_ids_observed` (before update) and the NEW set. Current implementation at line 233 fetches the existing session, so the old chip set is available.

**Write-back logic (simplified for the planner):**
```python
def _sync_coffee_flavor_notes(
    db: Session,
    *,
    coffee_id: int,
    old_session_ids: list[int],  # [] for new sessions
    new_session_ids: list[int],
) -> None:
    coffee = db.execute(select(Coffee).where(Coffee.id == coffee_id)).scalar_one()
    current = set(coffee.advertised_flavor_note_ids or [])
    added = set(new_session_ids) - set(old_session_ids)
    removed = set(old_session_ids) - set(new_session_ids)
    # Union adds, subtract removes (last-write-wins per D-09)
    updated = (current | added) - removed
    db.execute(
        update(Coffee)
        .where(Coffee.id == coffee_id)
        .values(advertised_flavor_note_ids=sorted(updated), updated_at=func.now())
    )
    # Caller commits
```

### Draft Restore Union-Merge (D-10)

`app/routers/brew.py:695`: `server_draft = brew_drafts_service.get_draft(db, by_user_id=user.id)` is fetched, then passed as context to the form template.

### CSV Import Write-Back (D-11)

In `import_brews` (`csv_io.py`), the current batch-commit loop at line 446-453 creates `BrewSession` objects and calls `db.commit()` once. After the commit, the write-back logic needs to iterate the pending sessions and call `_sync_coffee_flavor_notes()` for each. Since the batch is already committed, this should be a second transaction.

**Concern:** if the write-back fails after the sessions are committed, sessions exist but flavor-note sync is incomplete. D-09 last-write-wins means a re-import would fix it. For a household-scale import of 10-100 rows, a post-commit write-back loop is acceptable.

### Cross-User Concurrent Edit Risk (D-09)

The write-back uses `UPDATE coffees SET advertised_flavor_note_ids = <new_array>`. Two concurrent saves can race. The current `update_coffee()` in `services/coffees.py` does a full column overwrite. To handle concurrent races safely without introducing SERIALIZABLE transactions:

Use a Postgres array operation: `advertised_flavor_note_ids = array_cat(array_remove(advertised_flavor_note_ids, :removed_id), ARRAY[:added_ids])` expressed via `func` and `text()`. At household scale (two users, rare concurrent sessions), this is optional — D-09 explicitly accepts last-write-wins.

**Recommendation for planner:** Implement as a read-then-write (load current array, compute diff, write new array) within the same transaction as the session commit. This is sufficient for household scale. No advisory locks needed.

## Common Pitfalls

### Pitfall 1: Desktop Cancel targets wrong element

**What goes wrong:** Desktop Cancel button tries `hx-target="closest [data-row]"` but the form is rendered inside `#coffee-form-mount` (not inside any `[data-row]`). HTMX travels the DOM, finds no `[data-row]`, silently fails.
**How to avoid:** The Cancel button must use `hx-target="#coffee-form-mount"` with `hx-swap="innerHTML"` on the desktop path. The `form_target` variable injected by the router controls this.
**Warning signs:** Cancel button click does nothing; no visible error; DevTools shows a successful 200 response to `/coffees/empty-form` but no DOM change.

### Pitfall 2: Desktop edit success response fails to update the table row

**What goes wrong:** `update_coffee_handler` returns `coffee_row.html` (a `<tr>`) which HTMX tries to swap into `#coffee-form-mount`. The `<tr>` inside a `<div>` gets foster-parented — exactly the original "narrow column" bug.
**How to avoid:** Desktop path success response must use OOB swaps: one swap for the updated `<tr id="coffee-{id}">` targeting the row by ID, one swap to clear the form mount. Read `layout` from the form's hidden input.

### Pitfall 3: `extra="forbid"` on CoffeeCreate blocks new form fields

**What goes wrong:** Adding `origins_country[0]` style keys to the form payload hits `extra="forbid"` and causes a ValidationError rendered as a form error.
**How to avoid:** `_parse_form_payload` in the coffees router must collect and strip origin-related keys BEFORE passing to `CoffeeCreate`. Add them to `_NON_SCHEMA_FORM_KEYS` or handle them in `_parse_form_payload` like `advertised_flavor_note_ids`. The `origins` data goes directly to the service, bypassing Pydantic (similar to how `advertised_flavor_note_ids` is handled as a list of ints, not a sub-model).

### Pitfall 4: Alembic autogenerate detects wrong changes on the `coffee_origins` migration

**What goes wrong:** Autogenerate diffs the `CoffeeOrigin` ORM model against the DB. If the migration hasn't run yet, autogenerate generates a `create_table` statement. If the DB has the old columns (`country`, `origin`) but the model doesn't, autogenerate generates `drop_column`. This is correct — but the INSERT data-move step must be hand-added between the table create and the column drop.
**How to avoid:** Always autogenerate → review → add the INSERT step manually. Never run autogenerate-generated migration without reviewing the diff.

### Pitfall 5: `|tojson` in Alpine `data-*` attrs must use single-quoted attributes

**What goes wrong:** `data-initial-chips='{{ selected_flavor_notes|tojson }}'` works because `tojson` produces `"` characters that are safe inside single-quoted HTML attributes. Using double quotes breaks the attribute.
**How to avoid:** Any new Alpine `data-*` attribute that uses `|tojson` (e.g., for seeded origin rows or seeded varietal chips) MUST use single-quoted HTML attribute syntax. [VERIFIED: existing `coffee_form.html:209` uses this correctly — `data-initial-chips='{{ selected_flavor_notes|tojson }}'`]

### Pitfall 6: Service worker stale cache confounds UI verification

**What goes wrong:** After a Docker rebuild, the browser's service worker may continue serving stale fragments from cache. A desktop edit that appears broken may be using a cached pre-fix response.
**How to avoid:** Before UI verification, use Chrome/Edge DevTools → Application → Storage → "Clear site data" (not just cache bypass). The app's cache strategy uses SWR — content-deterministic versioning means a rebuild that changes templates WILL bump the cache key, but only if the build stage 1 COPY detects a change. [VERIFIED: project memory note `c9-sw-cache-content-deterministic`]

### Pitfall 7: htmx-indicator CSS not in tailwind.src.css

**What goes wrong:** If any new HTMX loading indicator uses `.htmx-indicator`, Tailwind's purge/JIT will not include it unless it appears in a scanned template or is added to `tailwind.src.css` as a safelist.
**How to avoid:** Add `.htmx-indicator { opacity: 0; transition: opacity 200ms; }` and `.htmx-request .htmx-indicator { opacity: 1; }` to `app/static/css/custom.css` if any new spinner uses this class. [VERIFIED: project memory note `strict-csp-blocks-htmx-indicator`]

### Pitfall 8: Image rebuild required for every template change

**What goes wrong:** The container is baked (no bind mount). A change to `coffee_form.html` does not take effect until `docker compose build coffee-snobbery && docker compose up -d coffee-snobbery`.
**How to avoid:** Always rebuild after template changes. During fast iteration, use `docker compose cp` for single-file updates: `docker compose cp app/templates/fragments/coffee_form.html coffee-snobbery:/app/app/templates/fragments/coffee_form.html` + container restart (not rebuild). [VERIFIED: CLAUDE.md]

```python
def create_varietal(db: Session, *, name: str, by_user_id: int, commit: bool = True) -> Varietal:
```

## Security Domain

CSRF is already enforced on all state-changing forms by `starlette-csrf` middleware. New endpoints (origin row template GET, varietal autocomplete GET, varietal create POST) follow the existing pattern:
- GETs for fragment templates: no CSRF required (read-only)
- POSTs for create/update: CSRF token via `<input type="hidden" name="X-CSRF-Token">` already in the base form template

No new authentication surfaces. No new AI or encryption surfaces. No changes to `app/services/encryption.py` or `app/services/scheduler.py`.
