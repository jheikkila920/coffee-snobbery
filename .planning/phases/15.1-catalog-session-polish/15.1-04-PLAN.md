<objective>
Implement bidirectional flavor-note sync between brew sessions and the parent coffee (D-06..D-12). On new session: prefill chips from parent coffee. On session save (create or update): diff/union the session's chips against the parent coffee's chips and write back. On CSV import: apply the same write-back. On draft restore: union the draft's chips with the coffee's current chips.

Purpose: Makes the chip-builder kettle-side experience accurate — the user sees what the coffee currently shows, and their refinements feed back to enrich (or correct) the catalog over time. Last-write-wins across users keeps the model simple.
Output: New `_sync_coffee_flavor_notes` helper in brew_sessions service; extended create + update + resolve_prefill; CSV import wired to the same write-back; draft restore union-merge in the brew router. Tests cover all four touch points.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client→POST /brew + POST /brew/{id} | User submits brew session; flavor_note_ids_observed is an integer list; FK validation via SQLAlchemy ensures referenced ids exist |
| client→POST /brew/import (CSV) | User uploads a brew-session CSV; same flavor-note write-back applies (D-11); CSRF protected; auth gated |
| client→draft restore | Server union-merges draft chips with current coffee chips; user sees additive view |
| brew_sessions._sync_coffee_flavor_notes | Cross-user mutation of shared catalog data; last-write-wins (D-09 accepted) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-15.1-22 | Tampering (cross-user catalog mutation) | _sync_coffee_flavor_notes write-back | accept (D-09) | Household trust model; CONTEXT explicitly accepts last-write-wins; tooltip in chip section documents the behavior to the user |
| T-15.1-23 | Tampering (concurrent race on advertised_flavor_note_ids) | Two simultaneous session saves for the same coffee | accept | Read-then-write within the same transaction is sufficient at household scale (RESEARCH "Cross-User Concurrent Edit Risk"); no advisory locks per D-09 |
| T-15.1-24 | Tampering (CSV row injects unknown chip ids) | csv_io import_brews | mitigate | Existing Phase 5 CSV importer validates flavor-note id existence; rows with unknown ids are rejected; the new write-back only runs on accepted rows |
