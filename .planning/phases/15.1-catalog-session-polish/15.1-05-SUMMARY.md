# Phase 15.1 Plan 05: Desktop Edit Form Layout (CATALOG-01) Summary

**One-liner:** Desktop edit form routing via dual Tailwind-responsive buttons and server-injected form_target/form_swap, with HTMX OOB swap for update success across all five catalog entities.

## What Was Built

CATALOG-01 (D-13/D-14/D-15/D-21) closes the desktop edit form width bug: previously, clicking Edit on a table row injected the form into `closest [data-row]` (a `<tr>` element), causing the form to foster-parent into a narrow table column. The fix routes desktop edits to the existing `#{entity}-form-mount` wide div instead.

**Pattern applied uniformly to coffees, roasters, equipment, flavor_notes, recipes:**

1. **Dual Edit buttons** in each row template â€” `md:hidden` (mobile, targets `closest [data-row]` with `outerHTML`) and `hidden md:inline-flex` (desktop, targets `#{entity}-form-mount` with `innerHTML` via `?layout=desktop`).

2. **Server-injected form_target/form_swap** â€” all five form templates consume `{{ form_target }}` and `{{ form_swap }}` from server context (no hardcoded targets). The `<form>` element and Cancel button both use these variables.

3. **Router layout param handling** â€” each router's edit GET handler accepts `layout: str | None = None`, sanitizes via T-15.1-29 (only `"desktop"` accepted), then computes `form_target`/`form_swap` and injects into template context. Update POST handlers read `layout` from form data before Pydantic validation and branch the response accordingly.

4. **OOB response on desktop update success** â€” `include_desktop_oob=True` adds `hx-swap-oob="outerHTML"` to the row `<tr>` element and appends a `<div id="{entity}-form-mount" hx-swap-oob="innerHTML"></div>` to clear the form mount. First-use of OOB for the update path in this codebase.
