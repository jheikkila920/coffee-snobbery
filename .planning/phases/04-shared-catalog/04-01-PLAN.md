**Part B: implement `app/services/photos.py`** (NEW)

Module docstring: "Photo pipeline: magic-byte verify → Pillow decode + re-encode → EXIF strip → resize → thumb. SEC-07 + SEC-4 (polyglot defense)." Mirror the structural template from `app/services/encryption.py` (pure primitives module, no DB writes).

Module-level constants per 04-RESEARCH.md §"Pattern 5":
- `PHOTOS_DIR = Path("/app/data/photos")`
- `MAX_BYTES = 5 * 1024 * 1024`
- `MAX_DECODE_PIXELS = 2000 * 2000 * 4` (16M)
- `MAIN_MAX_EDGE = 1600`
- `THUMB_MAX_EDGE = 400`
- `JPEG_QUALITY = 85`
- `JPEG_MAGICS = (b"\xff\xd8\xff\xe0", b"\xff\xd8\xff\xe1", b"\xff\xd8\xff\xee", b"\xff\xd8\xff\xdb")`
- `PNG_MAGIC = b"\x89PNG\r\n\x1a\n"`
- `WEBP_RIFF = b"RIFF"`, `WEBP_FOURCC = b"WEBP"`
- `_SAFE_FILENAME_RE = re.compile(r"^[0-9a-f]{32}(-thumb)?\.jpg$")` (D-06 path-traversal defense)

Set `Image.MAX_IMAGE_PIXELS = MAX_DECODE_PIXELS` at module import time (Pillow docs: it's a module-global; one assignment suffices).

Exception class: `class PhotoRejected(Exception)` — raised by `process_and_save` on any rejection; router catches and translates to 200 + form fragment per D-04.

Functions:
- `_verify_magic_bytes(head: bytes) -> str` — returns `"jpeg"|"png"|"webp"` or raises `PhotoRejected("Unsupported image format")`. Require `len(head) >= 12`.
- `process_and_save(raw_bytes: bytes) -> str` — full byte-flow per 04-RESEARCH.md Pattern 5 steps 1–10. ORDER MATTERS:
  1. `len(raw_bytes) > MAX_BYTES` → raise `PhotoRejected("Photo too large (max 5MB).")`.
  2. `_verify_magic_bytes(raw_bytes[:12])`.
  3. `with Image.open(BytesIO(raw_bytes)) as probe: probe.verify()` — separate decode pass.
  4. Re-open: `image = Image.open(BytesIO(raw_bytes)); image.load()` (verify() consumes the stream; re-open required per 04-RESEARCH.md Pitfall 2).
  5. `try: image.getexif().clear(); except Exception: pass` (belt-and-braces; primary EXIF defense is save-without-exif-kwarg).
  6. `image = image.convert("RGB")` if mode != "RGB".
  7. `image.thumbnail((MAIN_MAX_EDGE, MAIN_MAX_EDGE), Image.Resampling.LANCZOS)` — in-place resize.
  8. `filename_uuid = uuid.uuid4().hex`; save main to `PHOTOS_DIR / f"{filename_uuid}.jpg"` via `image.save(path, "JPEG", quality=JPEG_QUALITY, optimize=True)` — no `exif=` kwarg (primary EXIF strip).
  9. `thumb = image.copy(); thumb.thumbnail((THUMB_MAX_EDGE, THUMB_MAX_EDGE), Image.Resampling.LANCZOS)`; save thumb to `PHOTOS_DIR / f"{filename_uuid}-thumb.jpg"`.
  10. Return `f"{filename_uuid}.jpg"`.

  Wrap PIL exceptions (`UnidentifiedImageError`, `Image.DecompressionBombError`, generic `Exception` during decode) and translate to `PhotoRejected("We couldn't read this image. Try a JPEG, PNG, or WebP.")` with `log.warning("photo.decode_failed", error_class=type(exc).__name__)`. Create `PHOTOS_DIR` with `PHOTOS_DIR.mkdir(parents=True, exist_ok=True)` at the top of the function.

  Note on atomicity: per D-07 "write-new-then-delete-old", the canonical pattern would write to `.tmp` files then `os.replace`. At household scale and given the function does NOT delete an existing file (replace lives in a separate function below), direct save is acceptable. Document this trade-off in a comment.

- `replace_photo(old_filename: str | None, new_filename: str) -> None` — write-new-then-delete-old: new file is already on disk (from `process_and_save`); this function only unlinks the OLD file. If `old_filename is None`, no-op. Use `Path.unlink(missing_ok=True)` on both `{old_stem}.jpg` and `{old_stem}-thumb.jpg`. Log warnings on `OSError` but never raise.

- `unlink_safe(filename: str | None) -> None` — idempotent removal of `{stem}.jpg` + `{stem}-thumb.jpg`. Used by bags hard-delete (Phase 4) and orphan sweep.

- `sweep_orphans(db: Session) -> int` — STRICT ordering per D-07 + 04-RESEARCH.md Pitfall 9:
  1. Snapshot filesystem first: `on_disk = {p.name for p in PHOTOS_DIR.iterdir() if p.is_file() and p.suffix == ".jpg"}`.
  2. Query DB second: `rows = db.execute(select(Bag.photo_filename).where(Bag.photo_filename.isnot(None))).all()`; expand each `fn` to `referenced = {fn, f"{fn.removesuffix('.jpg')}-thumb.jpg"}`.
  3. Unlink the diff third. Return count. Emit `log.info(CATALOG_PHOTO_ORPHAN_SWEPT, count=count, total_on_disk=len(on_disk))`. Document the ordering in a comment that names the footgun.

- `_is_safe_photo_filename(name: str) -> bool` — returns `bool(_SAFE_FILENAME_RE.match(name))`. Exposed for `app/routers/photos.py` to import (plan 04-10).

Import: `from app.events import CATALOG_PHOTO_ORPHAN_SWEPT, CATALOG_BAG_PHOTO_UPLOADED, CATALOG_BAG_PHOTO_DELETED` at module top. `from app.models.bag import Bag` is required for the sweep query — import inside the `sweep_orphans` function body if circular-import risk surfaces (mirror the `app/dependencies/db.py` lazy-import pattern).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → multipart upload | Untrusted bytes from user (potentially: oversize, polyglot, decompression bomb, EXIF-laden, HEIC, HTML masquerading as image) |
| photos service → filesystem | Service must atomic-write to a trusted location and never accept user-controlled paths |
| sweep job → filesystem | List filesystem FIRST, query DB SECOND, unlink THIRD — reversing order destroys in-flight uploads |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-PHOTO | Tampering | `app/services/photos.py` filename generation | mitigate | UUID4 hex filenames; never user-controlled paths. `_is_safe_photo_filename` regex `^[0-9a-f]{32}(-thumb)?\.jpg$` defends serving route. |
| T-04-EXIF | Information Disclosure | `app/services/photos.py::process_and_save` | mitigate | Re-encode via `Image.save(..., "JPEG", quality=85)` without `exif=` kwarg; belt-and-braces `image.getexif().clear()` before save. |
| T-04-POLY | Tampering / Spoofing | `app/services/photos.py::process_and_save` | mitigate | Re-encode after Pillow decode strips trailing data per SEC-4; magic-byte gate before decode; serving route (plan 04-10) adds `Content-Type` explicit + `nosniff` + `Content-Disposition: inline`. |
| T-04-DOS | Denial of Service | `app/services/photos.py::process_and_save` | mitigate | `MAX_BYTES = 5MB` pre-buffer check (Content-Length + post-read len check); `Image.MAX_IMAGE_PIXELS = 16M` cap raises `DecompressionBombError` before allocation. |
</threat_model>
