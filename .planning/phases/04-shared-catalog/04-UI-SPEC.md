# Phase 4 — UI Design Contract (Shared Catalog)

> Visual and interaction contract for the five catalog surfaces (coffees, roasters, flavor notes, equipment, recipes), the bag photo upload pipeline, and the universal inline-expand + mini-modal patterns Phase 5+ will reuse. Generated by gsd-ui-researcher; verified by gsd-ui-checker.

This contract is prescriptive. The planner translates each section into plan-level tasks; the executor uses it as the visual source of truth.

## Design System

| Property | Value |
|----------|-------|
| Tool | none (no shadcn — Python/Jinja2/HTMX/Alpine stack, no npm pipeline per stack lock) |
| Preset | not applicable |

## Spacing Scale

Tailwind default 4px-multiple scale; declared values used in this phase. **Every spacing value in this contract is a multiple of 4. No `*.5` Tailwind tokens (`py-0.5`, `py-1.5`, `py-2.5`, etc.) anywhere.**

| Token | Tailwind class | Value | Usage |
|-------|----------------|-------|-------|
| 1 | `gap-1`, `p-1` | 4px | Icon-to-label gap inside chips, fine inline padding, flavor-note pill vertical padding |
| 2 | `gap-2`, `p-2`, `py-2`, `px-2` | 8px | Compact element spacing, input vertical padding, desktop secondary in-row button vertical padding |
| 3 | `gap-3`, `p-3`, `py-3` | 12px | Form field separation (Q4 Comfortable density), modal field gap, recipe-step per-step control vertical padding |
| 4 | `gap-4`, `p-4`, `px-4` | 16px | Step-builder row gap (Q4 — roomier than other forms), card body padding, button horizontal padding |
| 6 | `gap-6`, `p-6` | 24px | Section padding, list-page main container padding (mobile) |
| 8 | `gap-8`, `p-8` | 32px | Section breaks on desktop, modal outer padding |
| 12 | `py-12` | 48px | Page-level top/bottom padding (matches existing `pages/setup.html` rhythm) |

**Density rules (Q4 Comfortable):**
- Form fields: `gap-3` between adjacent fields, `py-2` on text inputs, `px-2` left/right (matches `setup.html` style).
- Recipe step builder rows: `gap-4` between steps, `py-3` on per-step controls (roomier so reorder + remove buttons clear 44px tap target; `py-3` + `text-base` line-height ≈ 48px ✓).
- Buttons primary: `py-2 px-4` → ~40-44px including border/text (verified ≥44px touch target with `text-base` 16px line-height-normal).
- Buttons secondary in-row: `py-2 px-3` on both desktop and mobile (uniform; ≥44px tap target with `text-base`).
- Card padding (mobile list rows): `p-4` (16px all around) so each card hits ≥44px clickable area.
- Flavor-note pills: `px-2 py-1` (8px / 4px) — `py-1` is the smallest declared vertical padding and stays on the 4-multiple scale.

**Exceptions:** none. All values are multiples of 4.

## Typography

Four-size scale. Two weights only.

| Role | Tailwind class | Size | Weight | Line height | Usage |
|------|----------------|------|--------|-------------|-------|
| Page heading (h1) | `text-2xl font-semibold` | 24px | 600 | `leading-tight` (1.25) | List page title ("Coffees", "Roasters", etc.); matches `pages/setup.html` h1. **No responsive size jump** — h1 stays at `text-2xl` across all breakpoints. |
| Section heading (h2) | `text-lg font-semibold` | 18px | 600 | `leading-snug` (1.375) | Coffee detail subsections ("Bags", "Brews"); modal title; equipment-type group headings |
| Body | `text-base` | 16px | 400 | `leading-normal` (1.5) | All form inputs, table cells, card body, paragraph copy. **Required** at 16px for MX-1 (already enforced globally for inputs). |
| Helper / label | `text-sm` | 14px | 400 (or `font-semibold` for emphasis on table column headers) | `leading-normal` (1.5) | Inline error text, field hint, "X bags" usage count, table column headers, flavor-note pills, mobile meta lines |

**Type scale (lock):** exactly **4 sizes — 14, 16, 18, 24px**. No `text-xs` (12px). No `text-3xl` (30px). No `md:text-3xl` responsive bumps.

**Weights used (lock):** exactly **2 weights** — `font-normal` (400) for body, `font-semibold` (600) for headings, table column headers, primary CTA labels, and the "+ Add step" button. **No `font-medium` (500), no `font-bold` (700), no `font-light` (300).** Variable-axis weights are out of scope.

**Font family:** Tailwind default `font-sans` resolves to `system-ui, -apple-system, "Segoe UI", Roboto, "Helvetica Neue", Arial, "Noto Sans", sans-serif, ...`. No `@font-face` declarations. No `link rel="preconnect"` to Google Fonts. Q3 lock.

**Numeric tabular alignment:** Recipe step builder cumulative-water + time columns use `tabular-nums` so digit columns line up without monospace.

## Inline-Expand Form Pattern (D-02)

The dominant CRUD pattern across all five entities. Form lives at the top of the list page as an expandable row; never a modal (modals are reserved for the cross-entity flow D-15) and never a separate page.

```
┌─ /coffees ──────────────────────────────────────────────┐
│  Coffees                                  [Add coffee]  │  ← h1 + accent CTA top-right
│                                                         │
│  [Filters: Roaster ▾  Country ▾  Process ▾  ☐ Archived]│  ← filter bar (coffees only)
│                                                         │
│  ─────────────────────────────────────────────────────  │
│  ┌─ inline form (hidden until [Add coffee] clicked) ─┐ │
│  │ New coffee                                  [×]    │ │  ← × top-right closes form
│  │ Name           [                       ]           │ │
│  │ Roaster        [autocomplete ▾  + Create new ]    │ │
│  │ Country        [                       ]           │ │
│  │ Process        [select ▾]                          │ │
│  │ Roast level    [select ▾]                          │ │
│  │ Flavor notes   [chips + autocomplete + Create new]│ │
│  │ Notes          [textarea                    ]      │ │
│  │                                                    │ │
│  │            [Cancel]  [Save]                        │ │  ← Cancel ghost, Save accent
│  └────────────────────────────────────────────────────┘ │
│                                                         │
│  Onyx — Geometry         Ethiopia · Washed   [Edit][Archive]
│  Heart — Kabingo         Burundi  · Natural  [Edit][Archive]
│  …                                                      │
└─────────────────────────────────────────────────────────┘
```

**Mechanics:**
- "Add coffee" button: `hx-get="/coffees/new"` → `hx-target="#coffee-form-mount"` → `hx-swap="innerHTML"`. The mount div sits between the filter bar and the first list row.
- Form fragment template extends nothing (it's a partial); root element is `<div class="rounded-lg border border-espresso-200 bg-cream-100 p-6 mb-4 dark:bg-espresso-900 dark:border-espresso-800">`.
- Form fragment includes the CSRF hidden input (line copies `pages/setup.html:10`).
- Save: `hx-post="/coffees"` → `hx-target="#coffee-list"` → `hx-swap="afterbegin"`. New row prepends to the list. The form fragment clears itself via `hx-swap-oob="innerHTML:#coffee-form-mount"` returning an empty div.
- Cancel: `hx-get="/coffees/empty-form"` → returns an empty fragment that replaces the form. No client-side toggle; server is the single source of truth for "is the form open?".
- × close button (top-right of form): identical to Cancel. **Accessible name:** `aria-label="Close new coffee form"` (parallel per entity).
- Validation error (D-04): server returns HTTP **200** with the form fragment re-rendered including the user's submitted values + inline field-level errors. NOT 422 (HTMX swaps cleanly only on 2xx). The errored form fragment replaces the open form in place.
- Edit: clicking "Edit" on a list row fires `hx-get="/coffees/{id}/edit"` → `hx-target="closest tr"` (desktop) / `closest [data-row]` (mobile card) → `hx-swap="outerHTML"`. The row becomes an editable form inline. Save → server returns the updated row fragment (which swaps in place).
- Archive: `hx-post="/coffees/{id}/archive"` confirms via inline `hx-confirm="Archive coffee — stops appearing in lists but keeps brew history."` (Q2 copy embedded in the confirm dialog).

**Loading state:**
- HTMX `htmx-indicator` class on a small SVG spinner placed inside the Save button. Button text replaced with spinner + "Saving…" while in-flight. Button is `disabled` and `aria-busy="true"` during the request.
- Global HTMX request class: `htmx-request:opacity-60` on the form root so the entire form dims while submitting.

**Width:**
- Mobile (<640px): form spans full content width (`px-6` page padding inherited).
- Desktop (≥768px): form has `max-w-2xl` (~672px) so it stays scannable; longer inputs (Notes) span full width inside the form.

## Mini-Modal Pattern (D-15) — THE ONLY MODAL IN PHASE 4

Used ONLY for "+ Create new roaster" and "+ Create new flavor note" launched from inside the coffee form's autocomplete dropdowns. This is the only modal shipped in Phase 4; the general modal pattern (MOB-08) lands in Phase 11.

**Responsive presentation:**
- **Mobile (<640px):** full-screen sheet. Slides up from bottom; covers viewport. Status bar respected via `pt-[env(safe-area-inset-top)]`. `bottom: env(safe-area-inset-bottom)` on the footer action row.
- **Desktop (≥640px):** centered dialog. `max-w-md` (~448px) wide, `max-h-[calc(100vh-4rem)]`, vertically centered.

**Layout:**

```
Mobile (<640px) — full sheet              Desktop (≥640px) — centered dialog
┌────────────────────────────────────┐    ┌──────── backdrop (bg-espresso-950/50) ──────┐
│ ▔▔▔▔ drag indicator (optional)     │    │                                              │
│                                    │    │     ┌──── modal (max-w-md) ───────┐         │
│ New roaster              [×]       │    │     │ New roaster           [×]   │         │
│                                    │    │     │                              │         │
│ Name           [           ]       │    │     │ Name        [           ]    │         │
│ Location       [           ]       │    │     │ Location    [           ]    │         │
│ Website        [           ]       │    │     │ Website     [           ]    │         │
│ Notes          [           ]       │    │     │ Notes       [           ]    │         │
│                                    │    │     │                              │         │
│ ─────────────────────────────────  │    │     │ ───────────────────────────  │         │
│         [Discard]  [Save]          │    │     │       [Discard]  [Save]      │         │
└────────────────────────────────────┘    │     └──────────────────────────────┘         │
                                          └──────────────────────────────────────────────┘
```

**Behavior:**
- Backdrop: `fixed inset-0 bg-espresso-950/50 dark:bg-espresso-950/70 z-40`. Click backdrop → close without saving (Q2 D-15).
- Modal panel: `fixed bottom-0 inset-x-0 sm:bottom-auto sm:inset-auto sm:top-1/2 sm:left-1/2 sm:-translate-x-1/2 sm:-translate-y-1/2 z-50 bg-cream-50 dark:bg-espresso-900 rounded-t-2xl sm:rounded-lg shadow-xl max-h-[90vh] overflow-y-auto`.
- Modal header: title `text-lg font-semibold` (h2), close button `×` top-right (`w-11 h-11` ≥44px tap target, `text-espresso-900 hover:bg-cream-100`). The `×` glyph is decorative; the button carries `aria-label="Close {entity} dialog"` (e.g., `Close new roaster dialog`) so screen readers announce a verb + object, not the glyph.
- Modal body: same form pattern as inline-expand; `p-6 sm:p-8` padding; field gap `gap-3`.
- Modal footer: sticky bottom inside the panel, `border-t border-espresso-200`, ghost "Discard" + accent "Save" buttons right-aligned with `gap-3`. On mobile, action row sticks to viewport bottom via `pb-[env(safe-area-inset-bottom)]`.
- ESC key: closes without saving. Alpine listener inside the modal component.
- Focus trap: lightweight — focus moves into the first field on mount; ESC + backdrop close return focus to the launching autocomplete option. Full focus-trap (tab cycling) is **not** required at household scale per CONTEXT discretion note; browser default tab order is acceptable for v1.
- Z-index stack: backdrop `z-40`, panel `z-50`. No other Phase 4 z-index reservations.
- Roaster modal fields: `name` (required, `text-base`), `location` (optional, `text-base`), `website` (optional, `inputmode="url"`, validated via Pydantic `HttpUrl` server-side), `notes` (optional `<textarea rows="3">`).
- Flavor-note modal fields: `name` (required), `category` (required `<select>` with the 9-value enum: fruit, floral, sweet, chocolate, nutty, spice, savory, fermented, other). Native `<select>` per MOB-07 pattern.
- Submit flow: POST with `as_modal=true` form field. Server returns `HX-Trigger: roaster-created` or `flavor-note-created` header with JSON payload `{id, name}` + an empty fragment that closes the modal. Parent coffee form's Alpine listener for that event sets the hidden `roaster_id` input and updates the visible label (D-16).
- Discard click: Alpine confirms inline ONLY if any field is dirty: "Discard unsaved changes?" using a tiny inline confirm pattern (no nested modal). If clean, closes immediately.

**Implementation:** Alpine component `Alpine.data('miniModal', factory)` registered in `app/static/js/alpine-components/mini-modal.js`. Single instance mounted globally in `base.html` (or just on catalog pages) at `#modal-mount` div. The launching control sends `hx-get` to fetch the modal body, which targets `#modal-mount > .modal-body`. The Alpine component reads `x-show` from a `data-open` flag toggled by an `htmx:afterSwap` listener.

## Autocomplete Dropdown (D-13, D-14)

Used on the roaster and flavor-note fields inside the coffee form (and on flavor-notes-observed in Phase 5 BREW-03; Phase 4 ships the pattern).

**Trigger:** `hx-trigger="input changed delay:350ms[target.value.length >= 2]"` + `hx-sync="this:replace"` (cancels in-flight on new keystroke — HX-4 pattern verbatim).

## Form Validation Errors (D-04)

Re-rendered inline form fragment with field-level errors. HTTP 200 status (not 422).

**Per-field error treatment:**
- Errored field: input gets `border-red-300 focus:ring-red-700 focus:border-red-700` (replaces default `border-espresso-200`).
- Error helper text below the input: `<p class="mt-1 text-sm text-red-700 dark:text-red-300">{message}</p>` — matches existing `pages/setup.html:7` palette (red-700) and adds `text-sm` for hierarchy.
- ARIA: `aria-invalid="true"` on the input, `aria-describedby="{field}-error"` pointing to the error `<p>` with id `{field}-error`.

**Form-level errors (non-field, e.g., "Roaster name already exists"):**
- Banner at top of form: `<div class="rounded-md border border-red-300 bg-red-50 px-4 py-3 text-sm text-red-700 dark:bg-red-950/30 dark:border-red-700/40 dark:text-red-300">{message}</div>`.
- The banner appears between the form heading and the first field.

**Submitted values preserved:** every field's `value` attribute (or `selected` for selects, `checked` for checkboxes) reflects the user's submission. Pydantic `ValidationError` is caught server-side; the validated-but-failed model dump is passed back into the template.

**No JS toast.** All errors are inline, server-rendered. This mirrors how `pages/setup.html` and `pages/login.html` already handle errors (`{% if error %}<p class="mt-4 text-red-700">{{ error }}</p>{% endif %}`) and extends it to field-level granularity.


- Step builder column: `md:basis-2/3`.
- Pour timeline column: `md:basis-1/3 md:ml-6`. Sticky inside the form so it stays visible while scrolling steps.

**Layout — mobile (<768px):** steps stack vertically; pour-timeline preview moves BELOW the steps (not alongside), full-width.

**Step row controls:**
- Per-step control vertical padding: `py-3` (12px) so reorder + remove tap targets stay ≥44px in concert with `text-base` line-height.
- Reorder mechanism: **up/down arrow buttons** (NOT drag handle). Two reasons: (1) drag-and-drop on touch is fiddly to implement accessibly; (2) up/down is keyboard-friendly out of the box. Each arrow is a 44×44 tap target (`w-11 h-11`) with `aria-label="Move step up"` / "Move step down". First step's "up" is disabled; last step's "down" is disabled. The `⋮⋮` glyph shown in the sketch above represents two stacked arrows.
- Remove `[×]` button: 44×44 tap target (`w-11 h-11`), top-right of the row, `text-espresso-600 hover:text-red-700`, `aria-label="Remove step {index}"`.
- Step label input: free-text `text-base` placeholder "e.g., Bloom, Main pour".
- Water input: `inputmode="decimal"` integer 0-2000, validated server-side by Pydantic.
- Time input: visible as `mm:ss` but stored as `time_seconds` integer. Use a `text` input with pattern `\d+:\d{2}` + Alpine parser. Validates 0-3600 server-side.
- Δ readout: Alpine-computed (`+50g · +0:45`); `text-sm text-espresso-600` directly below the inputs.

**"+ Add step" button:** `border border-espresso-300 px-4 py-2 rounded-md text-base font-semibold hover:bg-cream-100 inline-flex items-center gap-2`. Inserts a new step pre-filled with `water_grams = (previous water + 50)`, `time_seconds = (previous time + 45)`, `label = ""`. (Weight is `font-semibold`, the same weight used for primary CTA labels — keeps the two-weight contract intact.)

**Footer readout:** `Total water: {sum}g · Total time: {mm:ss}` — `text-base font-semibold` left-aligned, sits above the form actions. Alpine-computed.

**Submit:** the step array serializes into a single hidden input `<input type="hidden" name="steps" :value="JSON.stringify(steps)">`. Server-side Pydantic validates the array via a `StepSchema` with `water_grams: int = Field(ge=0, le=2000)`, `time_seconds: int = Field(ge=0, le=3600)`, `label: str = Field(max_length=80)`.

**Validation errors:** if a step is invalid (e.g., time goes backward), server returns the form fragment with the offending step highlighted (`ring-1 ring-red-300`) and inline error helper text below the bad field.

**Empty state:** new recipe form pre-loads with one empty step ("Bloom" label, water 0, time 0:00) so the user has a starting point.

## Pour-Timeline Preview (D-11)

Vertical bar visualization computed reactively from the `steps` array.

**Construction:**
- Container: `<div class="flex flex-col rounded-md border border-espresso-200 overflow-hidden" :style="{ height: totalSeconds * 4 + 'px' }">` — height is proportional to total brew time, 4px per second, clamped to min 200px / max 600px so a 30-second brew is still visible and a 6-minute brew doesn't blow out the page.
- Each step is a `<div class="flex-grow-0 flex items-center gap-2 px-3 text-sm" :style="{ height: step.delta_seconds * 4 + 'px', backgroundColor: ... }">`.
- Segment colors: **alternating shades** of the espresso ramp for visual rhythm — odd steps `bg-espresso-700`, even steps `bg-espresso-500`. Both render `text-cream-50` for label legibility. (Single accent color reads as "one event"; alternating reads as "distinct steps" without introducing rainbow colors.)
- Inside each segment: step label (left) + cumulative water at end of segment (right). Format: `Bloom        50g · 0:45`.
- Cumulative water labels sit at the **bottom** edge of each segment (representing the water level reached AT the end of that pour duration).

**Mobile (<768px):** same vertical bar; full width below the step list. Narrow segments (very short steps) still legible because labels collapse to just cumulative water if `delta_seconds * 4 < 32px`.

**Empty / single-step state:** if total time is 0 (new recipe), the preview shows an empty rounded rectangle placeholder with text "Add a step to see the pour timeline" (`text-sm text-espresso-600`, centered).

**Computation:** all in Alpine, no server round-trip. Step deltas computed as `step[i].time_seconds - step[i-1].time_seconds` (or `step[0].time_seconds` for the first step).
