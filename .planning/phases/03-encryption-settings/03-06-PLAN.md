<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Test process → `_multi_fernet` module singleton | Tests need to substitute fresh Fernet keys per test without polluting other tests. The pattern is `monkeypatch.setattr` on `app.config.settings.APP_ENCRYPTION_KEY` + a force-rebuild of `_multi_fernet` (importlib reload OR a dedicated test-only helper). |
| Test process → DB | DB-touching tests use the sync `SessionLocal()` and the `_postgres_reachable` probe from conftest. Plan 02 already established this pattern. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-T1 | Information disclosure | Test code logs or asserts on the raw key value | mitigate | Tests use `Fernet.generate_key()` ephemeral keys; assertions check round-trip equality of arbitrary plaintexts (`b"snobbery-test"`), never the key. The log-capture tests (`structlog.testing.capture_logs`) assert on the absence of `key=` and `ciphertext=` in captured events. |
| T-03-T8 | Audit log redaction completeness | Tests inadvertently introduce a sensitive field name that bypasses the redactor | mitigate | Tests assert on the documented fields only (`provider`, `last_four`, `model_name`, `user_id`, `setting_key`, `old_value`, `new_value`, `value_type`, `error_class`, `row_count`, `during`). Any new field name in a future test must be added to the redactor's `SENSITIVE_KEYS` review. |
</threat_model>
