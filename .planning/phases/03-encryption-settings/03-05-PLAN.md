<objective>
Wire the three Phase 3 lifespan hooks into `app/main.py` in the locked D-16 order: `encryption.startup_check()` (fail fast, no DB), then inside a sync `SessionLocal() as db` block `credentials.rewrap_if_needed(db)` (rotate ciphertexts if `APP_ENCRYPTION_KEY` changed), then `settings.prewarm_cache(db)` (populate the in-process cache so request handlers do zero-I/O reads).

Purpose: Make the encryption + settings substrate active at container start. Without this plan, Plans 03-02/03/04 ship code that's never invoked; with this plan, every container restart performs the sentinel round-trip → auto-rewrap-if-needed → prewarm cache.

Output: One small edit to `app/main.py` (three new calls inside the existing `lifespan` body + three import additions). No new module, no new lifespan; extend the existing async context manager.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Operator → container start | Lifespan runs once per container. Bad `APP_ENCRYPTION_KEY` flips healthcheck unhealthy (D-13 → docker-compose orchestration). |
| Lifespan → DB | The single `with SessionLocal() as db` block holds two service calls. Both commit independently inside their own functions. Order matters: rewrap MUST run before prewarm so the post-rewrap fingerprint lands in the cache (CONTEXT D-16 lock). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-T3 | Denial of service | Bad `APP_ENCRYPTION_KEY` causes startup to silently succeed and produce un-decryptable writes | mitigate | `startup_check()` runs FIRST inside lifespan (before any DB I/O) per D-16. On `EncryptionStartupError`, the exception propagates out of the async lifespan context manager, uvicorn exits with non-zero status, and the docker-compose healthcheck flips the container unhealthy. Operator sees the chained exception cause in the log stream. |
| T-03-T5 | Race / Ordering | `prewarm_cache` runs before `rewrap_if_needed`, causing the cache to hold the pre-rewrap fingerprint while the DB row holds the post-rewrap fingerprint — until the next cache invalidation | mitigate | D-16 hard-locks the order: rewrap_if_needed → prewarm_cache. PATTERNS.md analog enforces. The reverse order would briefly serve stale fingerprint reads. |
</threat_model>

    # Phase 3 hooks (D-16). All three are sync; await-free is fine inside the
    # async lifespan body. startup_check runs FIRST so a bad APP_ENCRYPTION_KEY
    # fails fast before any DB I/O. rewrap_if_needed runs BEFORE prewarm_cache

    **Why this exact placement** (PATTERNS.md citation): the three calls are inserted after the Phase 0 SELECT-1 invariant (proves DB reachable) and BEFORE the `log.info("app.startup", ...)` event.     **Locked constraints:**
    - DO NOT add `await` to any of the three calls. They are sync (D-07 + D-11). Wrapping in `await asyncio.to_thread(...)` would add overhead with no benefit at startup (lifespan is single-threaded by definition).
    - DO NOT split the three calls across multiple `SessionLocal()` blocks. One context manager wraps both DB-touching calls per PATTERNS.md analog.
    - DO NOT add a try/except around `encryption_startup_check()`. The exception MUST propagate out of lifespan so uvicorn exits non-zero. If the operator wants to recover, the path is "fix `APP_ENCRYPTION_KEY` and restart the container" — there is no in-process recovery.
    - DO NOT add a `try/except` around the `with SessionLocal()` block to "keep the app up if rewrap fails". Per Pitfall 7 (RESEARCH.md), the per-row try/except is INSIDE `rewrap_if_needed`. Outer try/except here would swallow legitimate DB errors (e.g., connection failure, lock timeout).
    - DO NOT reorder the three calls. D-16 is locked: startup_check → rewrap_if_needed → prewarm_cache.
    - DO NOT touch the shutdown half of lifespan (the lines after `yield`). Phase 3 has no shutdown hooks.
    - The existing `SELECT 1` smoke check stays — it's the Phase 0 invariant that the DB is reachable before any other DB call.
