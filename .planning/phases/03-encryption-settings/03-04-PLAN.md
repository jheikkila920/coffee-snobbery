<deviation_note>
## Intentional Deviation: Inline UPDATE of `app_settings.encryption_key_primary_fingerprint`

Both `set_provider_credential` (first-credential-set baseline) and `rewrap_if_needed` (post-rotation write) write the `encryption_key_primary_fingerprint` row in `app_settings` via a **direct `db.execute(update(AppSetting)...)`**, bypassing `app.services.settings.set_setting()`. This is a documented exception to the "all app_settings writes go through set_setting" convention. Two reasons:

1. **Atomicity** — The fingerprint write must land in the **same transaction** as the credentials UPDATE (set) or the rewrap loop (rotation). Calling `settings_service.set_setting` would commit a separate transaction, breaking the "credentials + fingerprint move together" invariant. A torn write here would leave the system in a state where the fingerprint claims the wrong primary key, masking a real rotation event from `rewrap_if_needed`'s idempotency check.

2. **`value_type` transition** — The seeded row from Plan 03-01 has `value_type='null'` (per CONTEXT.md `<specifics>`). The first fingerprint write needs to transition `value_type` from `'null'` to `'string'`. `set_setting` does NOT change `value_type` (immutable invariant per Plan 03-03 Task 1 acceptance criterion). The inline UPDATE writes both `value` AND `value_type` in a single statement, handling the one-time transition cleanly.

**Audit signal:** The inline UPDATE does NOT emit `admin.app_setting_changed`. The audit signals for fingerprint writes are `admin.api_credential_set` (when written as a side-effect of credential set) and `encryption.rewrap_completed` (when written as a side-effect of rewrap). This is the documented exception — fingerprint mutation is observable through the parent operation's audit event, not through a standalone `app_setting_changed` event.

**Cache consistency:** Both call sites invoke `settings_service.invalidate("encryption_key_primary_fingerprint")` after the inline UPDATE so the next `get_str` call re-reads from the DB (write-through invalidation, same contract as `set_setting`).

This is the only location in the codebase that writes to `app_settings` outside of `set_setting`. Future contributors writing other settings keys MUST use `set_setting`; the fingerprint row is uniquely exempt for the two reasons above.
</deviation_note>
