# Phase 16 — UI Design Contract

> Visual and interaction contract for the Cafe Quick-Rate phase (CAFE-01..CAFE-06). Generated by gsd-ui-researcher, verified by gsd-ui-checker. The shipped Snobbery design system (Tailwind v3 + custom palette + Phase 4/5/15.1 patterns) is the source of truth; this spec describes only what Phase 16 adds or extends, in those same terms.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (no shadcn — server-rendered Jinja + HTMX + Alpine, no npm) |
| Preset | not applicable |
| Component library | none (hand-rolled Jinja fragments + Alpine.data() factories) |
| Icon library | Heroicons via inline SVG (no font library; no external dep) |
| Font | system stack (Tailwind default: `ui-sans-serif, system-ui, ...`) |
| CSS pipeline | Tailwind **v3** standalone CLI binary, compiled at image build (Dockerfile stage 1). `darkMode: 'selector'`, `.dark` class on `<html>`. Source: `app/static/css/tailwind.src.css`. NEVER v4. |
| CSP posture | nonce-strict. Any new style class must compile in via tailwind.src.css. Never `<style>` blocks, never inline `style=` attrs except numeric width/height where unavoidable (existing `_cold_start.html` precedent) |

Mobile-first hard rule: every surface declared here renders correctly at 375px before being declared done. Bottom nav `<768px`, top nav `≥768px`.

## Spacing Scale

Tailwind v3 default 4-px scale. Phase 16 uses the existing tokens — no new ones.

| Token (Tailwind) | Value | Phase 16 Usage |
|------------------|-------|----------------|
| `1` (xs) | 4px | Icon-to-text gap inside chips, half-step nudges (avoid otherwise) |
| `2` (sm) | 8px | Compact pill / chip internal padding, inline icon gap (`gap-2`) |
| `3` | 12px | Card body internal vertical rhythm (between heading and meta lines) |
| `4` (md) | 16px | Default form-field gap, card padding, button padding `px-4 py-2` |
| `6` (lg) | 24px | Page outer padding (`px-6`), section gap |
| `8` | 32px | Major section breaks inside the form page |
| `12` | 48px | `py-12` page top/bottom (matches `pages/sessions.html` + `pages/brew_form.html`) |
| `14` | 56px | Rating star size (`w-14 h-14`) — reused, not redeclared |
| `16` | 64px | Reserved for bottom-nav clearance via `content-nav-safe-area` (existing rule) |

**Touch-target minimum:** `min-h-[44px] min-w-[44px]` on every tappable control on mobile (Apple HIG / WCAG 2.5.5). Matches the existing nav, coffee row Edit, session row Edit conventions.

**Exceptions:** none. Phase 16 introduces no new spacing token.

## Color

The existing two-ramp Snobbery palette (espresso = ramp-brown, cream = ramp-off-white) is the dominant + secondary system; Phase 16 introduces no new ramp. It pulls **amber-500** from Tailwind's stock palette as the **cafe-distinction accent** (D-07). Amber is already in app use for rating-star fill (`amber-400`) and AI advisory copy (`amber-600` light / `amber-400` dark) — using `amber-500` for the cafe border keeps the semantic family unified while distinguishing from those existing uses.

### 60 / 30 / 10 contract — light mode

| Secondary (30%) | `bg-cream-100`, `border-espresso-200` | `#F4EFE6`, `#C6AB94` | Cards, form container, nav strip, filter bar |

### Accent reserved for (explicit list — never general "interactive elements")

**`espresso-700`** (primary accent) reserved for:
1. The "Quick rate" header button on `/brew`
2. The "Save" button on `/cafe-logs/new` and `/cafe-logs/{id}/edit`
3. Active "Sessions" tab underline (`border-b-2 border-espresso-700`)
4. Progress bar fill on cold-start meter (existing — unchanged by Phase 16; meter shows combined brew+cafe count per D-15)

**`amber-500`** (cafe-distinction accent) reserved for:
1. `border-l-2 border-l-amber-500` (light) / `dark:border-l-amber-400` on the cafe card and the cafe desktop row (mobile-card pattern lifts coffee_row.html structure)
2. Inline coffee-cup SVG icon stroke on the cafe card top-left
3. Active "Cafe tastings" tab bottom border (`border-b-2 border-amber-500` / `dark:border-b-amber-400`)
4. Nothing else. Not used for any text, button, link, or focus ring.

**`red-700` / `red-400`** (destructive / error) reserved for:
1. Pydantic `ValidationError` field messages re-rendered into the form
2. `border-red-300` on the offending `<input>` when an error exists
3. The Delete confirmation step copy
4. Nothing else.

### What NOT to add

- No `border-l-amber-500` on anything that is not a cafe log surface.
- No `text-amber-*` on cafe-card body text (use existing `text-espresso-*` ramp for legibility; amber is reserved for the accent edge + icon only).
- No new color ramp. No `text-blue-*`, `text-green-*`, `text-purple-*`.
- No Alpine-driven dynamic class injection that swaps these tokens at runtime.

   ```
   M3 8h13v6a4 4 0 01-4 4H7a4 4 0 01-4-4V8zm13 1h2a2 2 0 010 4h-2
   ```

   `aria-label="Cafe tasting"` for screen-reader discoverability.

## Interaction Contracts

### Form field — required + autofocus (CAFE-01)
- `<input name="cafe_name" required autofocus>` AND server returns 200 + re-render with `errors.cafe_name = "Coffee name is required."` if empty (server is the trusth; `required` is a hint).
- Autofocus is the `autofocus` attribute first; for create mode the page-level Alpine handler also calls `.focus()` on `init()` as a belt-and-suspenders against safari ignoring the attribute on subsequent navigations.

### Rating control (CAFE-01)
- Reuse the existing `ratingStars` Alpine scope verbatim (`brew_form.html` lines 112–164). Same touch target (`min-h-[56px]`), same `w-14 h-14` star size, same half-step zones, same `amber-400` fill, same `Clear` button, same `tabular-nums` echo text.
- No new component, no new CSS, no new states. Just mount.

### Autocomplete dropdowns (CAFE-02)
- Roaster: existing pattern — `hx-get="/roasters/list"`, `entity="roaster"`, `create_new_endpoint="/roasters/new-modal"`. Dropdown supports `+ Create new` modal (existing miniModal scope at `#modal-mount`). Roaster created via this path becomes the FK target on submit.
- Flavor notes: existing pattern — `hx-get="/flavor-notes/datalist"`, chip-input via `observedFlavorNotes` (or a renamed cafe-scoped copy if naming hygiene wins out — planner's call; both work identically).
- Origin country: NEW pattern — same `autocomplete` Alpine scope, hx-get to new endpoint `/cafe-logs/origin-country-autocomplete`. **NO `+ Create new`** (D-03: free text, plain suggestions only). The dropdown shows up to 20 matches; if zero matches and ≥2 chars, the dropdown collapses and the user's typed value submits as-is.

### Tab toggle (D-06)
- Server-side `?tab=brew` (default) or `?tab=cafe`.
- Anchors carry `hx-get="/brew?tab=cafe"`, `hx-push-url="true"`, `hx-target="#session-list"`, `hx-swap="outerHTML"`. Without HTMX (no-JS fallback) the plain `href` still works.
- Active-tab style: `border-b-2` in espresso-700 (Sessions) or amber-500 (Cafe tastings). Inactive: muted text color via `text-espresso-600 dark:text-cream-300`. No icon on tabs.
- Filter state IS tab-scoped: switching tabs resets to no filters by re-rendering the filter form for the active tab only.
- Back/forward browser navigation correctly replays state (hx-push-url guarantee).
- **No Alpine.js for tab state.** CSP nonce + back-button correctness both favor server-side.

### Dual Edit button (CAFE-06 / D-21)
- Mobile (`md:hidden`): `hx-get="/cafe-logs/{id}/edit"`, `hx-target="closest [data-row]"`, `hx-swap="outerHTML"`. The form renders inline replacing the card.
- Desktop (`hidden md:inline-flex`): `hx-get="/cafe-logs/{id}/edit?layout=desktop"`, `hx-target="#cafe-form-mount"`, `hx-swap="innerHTML"`. The form renders into the mount above the list, list stays visible.
- Touch target: `min-h-[44px] min-w-[44px]` on the mobile variant. Desktop variant follows existing coffee row's smaller `px-3 py-1` (desktop has cursor, no 44px floor required).

### Delete cafe log (CAFE-06)
- Mobile + desktop both use the same Delete button next to Edit.
- HTMX shape: `hx-post="/cafe-logs/{id}"`, `hx-vals='{"_method": "DELETE"}'` (or hidden `<input name="_method" value="DELETE">`), `hx-confirm="Delete this cafe tasting? This cannot be undone."`, `hx-target="closest [data-row]"`, `hx-swap="outerHTML"`.
- Server returns 200 with empty body OR a 200 with an `hx-swap-oob` clearing the cafe-form-mount if the form was open editing this row. Mirror the existing brew session delete shape.

### Photo upload
- Reuse the existing `photo-upload.js` Canvas downscale handler (no change). Reuse the existing `app/services/photos.py` pipeline. The form `<input type="file" name="photo" accept="image/jpeg,image/png,image/webp">` — same as bag form.
- Failure mode: server returns 200 + form re-render with `errors.photo`. Existing pattern.

### Form Save (D-11 sticky)
- The Save button is `<button type="submit"` inside the form. The form posts to `/cafe-logs` (create) or `/cafe-logs/{id}` (update).
- Sticky behavior at 375×667 viewport: the rating control + Save button must both be visible without scroll when the page first renders. The page H1 (`text-2xl mb-6`) + coffee name field + rating (`min-h-[56px]`) + Save button fit within ~580px of vertical space — comfortably under 667. **Verify at planner time** with a Playwright assertion: navigate to `/cafe-logs/new`, viewport 375×667, assert the Save button's `boundingBox().y + height <= 667`.
- Sticky on mobile (`sm:sticky sm:bottom-0` pattern from brew_form's save bar) ensures the Save remains reachable above the bottom nav per MOBILE-04 (which Phase 21 owns but Phase 16 doesn't violate).

### Success / failure routing
- POST success → `HX-Redirect: /brew?tab=cafe` (returns user to the cafe list).
- POST validation error → HTTP 200 + re-render `pages/cafe_log_form.html` with the form errors populated. NO `HX-Redirect`. The page (or the form-mount in desktop edit) swaps in place — matches existing form-error UX.
- POST server error (500) → HTMX default (no swap); render a top-level error banner via the existing pattern (matches login/setup failures).

### Cold-start meter (D-15) — non-visual change to math
- Frontend renders identical bar + copy structure.
- `gate.sessions` is now `(brew_count + cafe_count)`; `gate.distinct_notes` is distinct flavor_note_ids across brew_sessions.flavor_note_ids_observed UNION cafe_logs.flavor_note_ids, rating ≥ 4.0.
- The progress percent calculation in `_cold_start.html` already caps each at the goal (3 + 5 = 8 units); D-15 only changes the inputs.

## Accessibility Contract

- All interactive controls reachable via keyboard (tab through form, Enter submits, Escape closes autocomplete dropdown — existing `autocomplete` scope handles this).
- Touch targets ≥ 44×44px on mobile (the `min-h-[44px] min-w-[44px]` floor used throughout).
- Form labels associate via `<label>` (existing brew_form pattern).
- Error messages live in `<p class="text-sm text-red-700">` immediately after the offending control AND `aria-describedby` points at the error id from the input (existing pattern — match it).

## Risks and Open Questions (surface to planner)

1. **Cafe-form chip widget naming** — D-11 says reuse the flavor chip pattern. The brew form's Alpine scope is named `observedFlavorNotes` and is currently mounted with `data-initial-chips`. The cafe form needs the same shape but the semantic name "observed" doesn't fit (cafe coffees aren't brewed). Two options for the planner:
   - **Option A (cheaper):** reuse `observedFlavorNotes` verbatim on the cafe form; document the slight semantic stretch in a code comment.
   - **Option B (cleaner):** rename to a generic `flavorNoteChips` factory + alias `observedFlavorNotes` to it for the existing brew form. Slightly more work, no behavior change.
   - Recommend Option A for Phase 16 to minimize risk of touching brew_form scope; Option B is a clean follow-up.

2. **Save button visibility at 375×667** — D-11 stipulates "Save sticky-bottom OR visible without scroll." The existing brew_form Save sits below several taller controls and uses the sticky pattern. For the cafe form, the only required-top fields are coffee name + rating, so Save SHOULD be visible without scroll even non-sticky. **Recommendation:** apply the same sticky pattern as brew_form to be safe at smaller-than-375 viewports and at landscape orientation. Planner should add a Playwright assertion (375×667 portrait) that the Save button's bottom edge falls within the first viewport.

3. **D-21 desktop mount placement on the Cafe tab** — Phase 15.1 D-13 placed `#coffee-form-mount` above the list. Mirror this: `<div id="cafe-form-mount"></div>` directly above the cafe list region, visible only on md+ (`<div class="hidden md:block" id="cafe-form-mount"></div>`). The brew tab does not need a `#brew-form-mount` (brew uses its own dedicated `/brew/new` page — no mid-page edit pattern). This is a cafe-tab-only addition to `pages/sessions.html`.

4. **Filter state on tab switch** — D-06 says filters are tab-scoped. When the user switches Sessions → Cafe tastings with brew filters set, those brew filters do NOT carry. The cafe tab renders with no filters active. Implementation note for the planner: the GET handler should NOT echo any non-cafe-filter query params back onto the cafe filter form, even if present in the URL. Document this in the handler.

5. **Cold-start meter copy wording** — D-15 changes the math but not the copy. Current copy says "Log N more brew(s)." If users ask for clarity, change to "Log N more brews **or cafe tastings**." Tracked as a potential follow-up; not a Phase 16 deliverable.
