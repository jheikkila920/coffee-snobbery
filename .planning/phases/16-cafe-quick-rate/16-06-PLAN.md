<objective>
Plan 16-06 closes Pitfall 8 (photo orphan sweep silent-data-loss landmine). The cafe vertical slice (plans 16-01..16-04) persists `cafe_logs.photo_filename` but the existing nightly orphan-sweep job in app/services/photos.py:sweep_orphans only knows about `bags.photo_filename`. Without this plan, every cafe photo would be deleted on the next nightly sweep — a silent data-loss bug that wouldn't surface until users started losing their cafe photos. This is a small, surgical, two-file plan that extends sweep_orphans by ~5 LOC + adds one mandatory regression test.

Purpose: prevent silent data loss for cafe photos uploaded via plan 16-02's POST /cafe-logs route. The sweep_orphans extension is small enough to fit in the same wave as the analytics work (plan 16-05) — both depend only on the plan 16-01 model + skip-gate, and they touch disjoint files (photos.py vs analytics.py + test_photos.py vs test_analytics.py), so they run in parallel.

Output: a one-block extension to photos.sweep_orphans (lazy CafeLog import + parallel SELECT + set-union into referenced_main) + the mandatory regression test test_sweep_keeps_cafe_photos.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| photos.sweep_orphans → filesystem | Trusted intra-process boundary — the worker is the app user; PHOTOS_DIR is the bind-mount; UUID4 filename pattern blocks path traversal upstream (photos.py:_SAFE_FILENAME_RE invariant). The sweep is a destructive operation (unlinks files); the correctness invariant is "no file unlinked while a DB row references it." |
| nightly scheduler → sweep_orphans | Trusted intra-process boundary — APScheduler invokes the sweep on its cadence. No external input crosses here. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-16-06-01 | DoS / Information Disclosure | photos.sweep_orphans silent data loss for cafe photos (Pitfall 8) | mitigate | Extend sweep_orphans to UNION cafe_logs.photo_filename into the referenced_main set. Single-SQL extension, no new sweep loop, preserves the FS-first / DB-second / unlink-third invariant. The mandatory regression test test_sweep_keeps_cafe_photos seeds a cafe photo + an orphan, runs sweep, asserts cafe survival. ASVS V12 (data integrity of user uploads). |
| T-16-06-02 | Tampering | Race condition: a cafe_log row created between FS-snapshot and DB-query | accept | The strict FS-first / DB-second / unlink-third ordering invariant from D-07 / Phase 4 already addresses this: any photo file added to disk AFTER on_disk is captured will not be in on_disk's iteration set, so it cannot be unlinked. Any DB row inserted AFTER the SELECT will not appear in referenced_main, but its photo file is also not on the FS yet (the cafe-log upload writes the file BEFORE committing the row, per app/routers/cafe_logs.py from plan 16-02). Effective window for the race is zero in practice. |
| T-16-06-03 | Information Disclosure | per-user scoping on sweep query | accept | sweep_orphans is a global cleanup job; it does NOT need per-user scoping. The query reads ALL cafe_logs.photo_filename values (all users) and ALL bags.photo_filename values (all users) to build the global referenced_main set. This is correct: a file is "referenced" if ANY row in either table points to it, regardless of user. Per-user filtering would CAUSE silent data loss for users whose photos are referenced by other users' rows (impossible today, but the global query is the safe shape). |
