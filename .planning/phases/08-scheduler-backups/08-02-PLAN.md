<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| backup.py → pg_dump subprocess | Connection credentials cross from settings.DATABASE_URL into a child process. Single-tenant container; no user input reaches this path. |
| backup.py → filesystem (/app/data/backups) | Backup files contain full plaintext DB data. Volume is reachable only inside the Docker network. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-03 | Information Disclosure | PGPASSWORD in subprocess env | accept | On Linux, per-process env is not visible to other users via `ps` by default and the container is single-tenant — this is the accepted pg_dump-in-Docker pattern (08-RESEARCH Security Domain). Acceptance criterion: PGPASSWORD passed via `env=` dict, never as a CLI arg and never logged. |
| T-08-04 | Tampering | subprocess injection via DATABASE_URL | mitigate | `subprocess.run([...])` with a LIST and NO shell=True — special chars in the URL cannot inject. Acceptance criterion in Task 1: `shell=True` absent from the file. |
| T-08-05 | Information Disclosure | pg_dump output is plaintext DB data | accept | Backups written only to the coffee_snobbery_backups volume on the Docker network; off-site/encrypted shipping is explicitly v2-deferred (CONTEXT <deferred>). |
| T-08-06 | Information Disclosure | credentials leaking into logs | mitigate | BACKUP_* events carry filenames/byte-counts/status only — never the password or the DATABASE_URL. Acceptance: no event field contains the password (grep: no `password` key in any backup.* log call). |
</threat_model>
