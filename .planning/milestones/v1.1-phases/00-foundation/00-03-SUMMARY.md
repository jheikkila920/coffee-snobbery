# Phase 0 Plan 03: SQLAlchemy + Alembic + Initial Migration — Summary

**One-liner:** SQLAlchemy 2.0 sync engine with the locked household-scale pool knobs, five `Mapped[...]` ORM models (User, Bag, WishlistEntry, AIRecommendation, AppSetting) with bags.coffee_id shipped FK-free for Phase 4 retrofit, complete Alembic wiring driven by `settings.DATABASE_URL`, and one hand-authored mega-migration that installs three Postgres extensions, creates five tables, and seeds 19 documented `app_settings` rows. Plus nine schema/seed introspection tests that skip cleanly when Postgres is unreachable so unit-only test runs stay green.

## What Was Built

Plan 00-03 ships the database layer in three atomic tasks:

1. **Engine + Models (Task 1)** — `app/db.py` constructs the SQLAlchemy 2.0 sync engine with the locked pool knobs (CONTEXT D-10 / PITFALL SH-2) and exposes `engine`, `SessionLocal`, `dispose_engine`, and a re-export of `Base`. Five SQLAlchemy 2.0 `Mapped[...]` model classes — one per module under `app/models/` — populate `Base.metadata` with the five tables. `app/models/__init__.py` re-exports every model so Alembic autogenerate (Phase 1+) sees the full schema. **The cost-observability column set lands NOW** on `AIRecommendation` (11 columns) — retrofitting columns onto a populated table is painful, and Phase 7 must not face that.
2. **Alembic + Initial Migration (Task 2)** — `alembic.ini` at repo root, `app/migrations/env.py` reading `settings.DATABASE_URL` (FOUND-10 compliance), and `0001_initial.py` running three `CREATE EXTENSION IF NOT EXISTS` statements FIRST (citext, pg_trgm, unaccent — FOUND-06; the citext type must exist before `users` is created), then creating the five tables in dependency order (users → bags → wishlist_entries → ai_recommendations → app_settings), then `bulk_insert`ing 19 documented seed rows into `app_settings` via an `sa.table()` shim (never import from `app.models` inside a migration). Downgrade drops the tables in reverse order but DOES NOT drop the extensions — they are cluster-shared and could break sibling databases.
3. **Schema Tests (Task 3)** — `tests/test_migrations.py` with nine integration tests that introspect `pg_extension`, `information_schema.columns`, `pg_indexes`, and the seed rows. A LOCAL `pg_session` fixture (NOT in `tests/conftest.py`) connects to a live Postgres via `settings.DATABASE_URL` and skips every test in the module if the connection fails — keeping unit-only `pytest` runs (Wave 0) green while letting `make smoke` (Plan 05) exercise the full live-DB path.

## Notes for Plan 00-04 (Dockerfile + entrypoint + app factory)

- **`entrypoint.sh` runs `alembic upgrade head` before `exec uvicorn`.** Migration is idempotent (`CREATE EXTENSION IF NOT EXISTS`, `CREATE TABLE` against an empty schema). Compose's `depends_on: condition: service_healthy` already gates this on Postgres `pg_isready` (CONTEXT D-07).
- **`app/main.py` lifespan opens `engine` and disposes on shutdown.** Importing `app.db` constructs the engine; the lifespan only needs to call `dispose_engine()` on shutdown:
  ```python
  from contextlib import asynccontextmanager
  from app.db import dispose_engine
  @asynccontextmanager
  async def lifespan(app):
      yield
      dispose_engine()
  ```
- **`/healthz` (CONTEXT D-08) uses `SessionLocal` + `SELECT 1`.** Pool timeout is already 5s in the engine config — `/healthz` will fail-fast on pool exhaustion. Suggested implementation:
  ```python
  from app.db import SessionLocal
  from sqlalchemy import text
  def healthz():
      with SessionLocal() as session:
          session.execute(text("SELECT 1"))
      return {"status": "ok"}
  ```
- **DO NOT import from `app.db` in `app/main.py`'s module-level scope** if you want lifespan-controlled engine lifetime — keep DB calls inside the route function bodies / lifespan. (The engine itself is constructed at module import; this is fine — Plan 02's logging is initialized before any DB use.)

## Notes for Future Phases

- **`app/models/__init__.py` is the canonical registry.** Every new model in any future phase MUST be re-exported here (`from app.models.foo import Foo`) AND added to `__all__`. Without that, Alembic autogenerate won't see the table.
- **Phase 4 adds the bags.coffee_id FK constraint.** The `tests/test_migrations.py::test_bags_coffee_id_has_no_foreign_key` test is a Phase-0-only assertion — Phase 4's plan will update this test (flip the assertion to `count == 1` and assert the FK targets `coffees.id`).
- **Phase 7 writes the cost-observability columns.** All 11 columns already exist with their NOT NULL / DEFAULT shape. Phase 7 just needs to populate them on every `INSERT INTO ai_recommendations`.
- **Migration policy** (CLAUDE.md "Things to never do silently"): never edit a committed migration. Add a new revision.
- **Pool knob policy** (CONTEXT D-10): never change the pool knobs in `app/db.py` without re-running the Phase 0 Plan 05 pool exhaustion smoke test.
