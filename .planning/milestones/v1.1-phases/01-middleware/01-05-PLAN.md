<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → app (state-changing) | Any POST/PUT/DELETE/PATCH from a logged-in user must be proven to originate from the app's own UI, not a third-party site |
| browser JS → CSP enforcement | Even compromised/extension-injected JS must not be able to bypass CSRF or run inline eval-style code |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Tampering | CSRF on state-changing forms | mitigate | `starlette-csrf` 3.0 double-submit-cookie: cookie + matching `X-CSRF-Token` header must both be present and match for POST/PUT/DELETE/PATCH. ASVS V4.2.1 (CSRF protection) + V4.2.2 (state-changing operations). |
| T-05-02 | Tampering | HTMX-driven state-changing requests | mitigate | `htmx:configRequest` listener attaches `X-CSRF-Token` from `<meta>` automatically — every HTMX POST inherits the protection without per-form boilerplate. RESEARCH §2 confirms the pattern is HTMX-2.x-native. |
| T-05-03 | Tampering | CSP bypass via HTMX `hx-on:`, `hx-vals='js:`, `hx-headers='js:`, or eval-using event filters | mitigate | `htmx.config.allowEval = false` as the first executable line of `htmx-listeners.js`; Plan 01's grep test bans the template-side patterns. Defense in depth — CSP also blocks via `'unsafe-eval'` omission. |
| T-05-04 | Tampering | CSP-report endpoint flooded with forged CSRF challenges if it were subject to CSRF check | mitigate | `exempt_urls=[re.compile(r"^/csp-report")]` — browsers POST violation reports with `application/csp-report` content-type and without app cookies; CSRF would always fail. (ASVS V4.2.1 exception for known-safe endpoints.) |
| T-05-05 | Information Disclosure | `csrftoken` cookie sniffable over HTTP | mitigate | `cookie_secure=True` enforces HTTPS-only transmission. Documented as a foot-gun in RESEARCH §3 — `False` even in local dev would silently downgrade production safety. |
| T-05-06 | Tampering | CSRF token rotation breaking HTMX fragment swaps (PITFALL HX-1) | mitigate | `starlette-csrf` 3.0 sets the cookie ONCE per session and does not rotate per response — verified by RESEARCH A1 assumption + Wave 0 `test_no_rotation`. This was the locked-in reason for choosing double-submit-cookie over rotated tokens (PROJECT.md row 15). |
| T-05-07 | Tampering | Future contributor swapping `starlette-csrf` for a rotated-token library | accept | Plan 10 ADR documents the double-submit-cookie decision; PROJECT.md row 15 already captured. Drift is detected at PR review, not via automated check. |
</threat_model>
