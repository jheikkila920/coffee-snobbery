<critical_implementation_note>
**regenerate() is `async def` but the job body is `sync def`.** APScheduler's
AsyncIOScheduler runs a sync job body in its ThreadPoolExecutor (off the event loop),
which is required because the job does sync DB work (08-RESEARCH Pattern 3, Pitfall 4).
But a sync function cannot `await` regenerate. The job body therefore bridges sync→async
PER USER via `asyncio.run(regenerate(uid, "scheduler", db=db, force=False))` — the SAME
pattern tests/conftest.py `_seed_user` already uses (`asyncio.run(_do())`) to call async
code from a sync context. Each `asyncio.run` opens and tears down a fresh event loop in
the worker thread; at household scale (sequential, one user at a time) this is correct and
low-cost. Do NOT make the job body `async def` (would run on the event loop and block it on
the sync DB reads). Do NOT call `regenerate` without awaiting/asyncio.run (it returns a
coroutine that never executes).
</critical_implementation_note>
