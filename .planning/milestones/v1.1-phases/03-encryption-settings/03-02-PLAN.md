<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Operator → env (`APP_ENCRYPTION_KEY`) | Operator provides comma-separated Fernet keys via the .env file or docker-compose `environment:`. A malformed or empty key is the trigger for D-13's fail-loud path. |
| App import path → `_multi_fernet` module-level singleton | Constructed once at first import; any failure here is a hard startup failure (the whole point of D-13). |
| App → log stream | Lifespan-emitted events (`encryption.startup_ok`) have no `request_id`; the structlog redactor in `app/logging.py` already covers `api_key`/`secret`/`encryption_key`-shaped keys (no new redaction needed here). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-T3 | Denial of service / Tampering | App boots with malformed `APP_ENCRYPTION_KEY` and silently produces ciphertext nothing can decrypt | mitigate | `_build_multi_fernet()` raises `ValueError` on empty key list; `Fernet()` raises `binascii.Error` on malformed base64; `startup_check()` performs `decrypt(encrypt(sentinel))` round-trip and wraps any `InvalidToken/ValueError/TypeError` in `EncryptionStartupError` chained via `__cause__` so uvicorn exits non-zero and the docker-compose healthcheck flips unhealthy. RESEARCH.md Pitfall 1 enforced. |
| T-03-T1 | Information disclosure | Module imports or constructs leak the raw `APP_ENCRYPTION_KEY` into log lines / error messages | mitigate | `EncryptionStartupError` message names the env var but NEVER includes the key value; `startup_check` log emit carries `fingerprint=primary_key_fingerprint()[:8]` (8-char prefix of SHA-256, not the key); structlog redactor already strips any field literally named `encryption_key`/`secret` (Phase 1 D-14); CLAUDE.md "never log API keys, passwords, or session tokens" reaffirmed |
| T-03-T8 | Audit log redaction completeness | A new event field name in Phase 3 bypasses the Phase 1 redactor | accept (verified) | Phase 1's `SENSITIVE_KEYS` already covers `api_key`, `api_key_encrypted`, `encryption_key`, `secret`. Plan 03-02 emits only `fingerprint` (8-char prefix) and the constant event name — no new sensitive field name added. Verification: `app/logging.py` SENSITIVE_KEYS set is read; no Phase 3 field shadows or evades it. |
